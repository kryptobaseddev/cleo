# ADR-079: Docs SDK Boundary Contract

- **Status**: Accepted
- **Date**: 2026-05-23
- **Tags**: docs, sdk, boundary, llmtxt, integration, contracts
- **Epic**: T10292 (E4-DOCS-SDK-BOUNDARY)
- **Saga**: T10288 (SG-DOCS-INTEGRITY)
- **Extends**: ADR-078 (Docs SSoT as DB-Backed Provenance Graph) — does NOT supersede
- **Cross-refs**: ADR-083 (Hierarchy + Sentient Substrate Frame)
- **Related**: T10353 (CLI verb matrix), T10354 (SDK import-edge surface),
  T10355 (gap verdicts — 0 SDK-owned, 8 CLEO-owned, 2 shared-CLEO-led)
- **Authors**: cleo-prime (2026-05-23)

## §1 Context

The `llmtxt` package — published from `https://github.com/kryptobaseddev/llmtxt`
and consumed via `npm` as `llmtxt` — is an external upstream SDK that CLEO uses
for primitive operations: identity (Ed25519 sign/verify), blob filesystem
layout, agent-session lifecycle, similarity ranking, knowledge-graph
construction, and version patch math. ADR-078 layered CLEO's docs SSoT
(provenance graph + supersession + update-in-place + slug routing) on top of
those primitives.

Wave 1 of Epic T10292 (E4-DOCS-SDK-BOUNDARY) audited the integration surface:

- **T10353 CLI verb matrix** (PR #578) catalogued 21 `cleo docs` + `cleo
  changeset` subcommands and traced each through `packages/cleo/src/dispatch/
  domains/docs.ts`, identifying which SDK primitive (if any) each invokes.
- **T10354 SDK import-edge surface** (PR #577) catalogued 11 static + 12
  dynamic `from 'llmtxt/*'` imports across `packages/{cleo,core,contracts}/
  src/`, classified each as type-only / function / extends, and flagged six
  new gaps (F-1 through F-5 plus F-8/F-9).
- **T10355 gap verdicts** (PR #586) classified all 10 known gaps —
  T10238 (unknown-flag silent absorb), T10153 (no ADR auto-numbering),
  T10167 (no similarity warn), T10294 (slug collision), and F-1..F-5/F-8/F-9 —
  as `CLEO`, `SDK`, or `shared` responsibility.

The verdict matrix concluded: **0 pure SDK gaps, 8 CLEO-only gaps, 2
shared-CLEO-led gaps (F-1 `AgentIdentity` re-export sprawl, F-2/F-3 cluster
where SDK types leak through CLEO's public envelopes).** Zero upstream issues
need to be filed against `kryptobaseddev/llmtxt`. Every gap's fix lives inside
CLEO's contracts, dispatch, CLI, or persistence layers.

That outcome is durable only if the boundary contract is written down. Without
a codified policy, future PRs will:

1. Re-export new SDK types from `packages/contracts/` or
   `packages/core/src/index.ts` (the F-1/F-2/F-3 failure mode),
2. Mis-classify gaps and file unnecessary upstream issues,
3. Add new static `from 'llmtxt/*'` imports that crash the runtime when the
   SDK is missing despite `optionalDependencies` promising graceful absence
   (the F-5 inconsistency),
4. Skip the `throwUnavailable` wrapper at new sites, re-introducing the
   degraded-mode split (F-4),
5. Drift on what "CLEO owns" vs "SDK owns" — re-litigating the matrix every
   six months.

This ADR codifies the contract so the verdict matrix survives future churn.

## §2 Decision

### §2.1 What CLEO MAY assume from llmtxt SDK

CLEO is entitled to assume:

1. **Semver stability per the published peer-dependency range.** Every
   `packages/*/package.json` that references `llmtxt` MUST pin a semver range
   that captures the major+minor surface CLEO consumes. Patch bumps are safe;
   minor bumps require regression review; major bumps require an ADR
   amendment.
2. **Primitive correctness.** `AgentIdentity` (Ed25519 sign/verify),
   `BlobFsAdapter` + `hashBlob` (content-addressed storage),
   `rankBySimilarity` (vector similarity), `buildGraph` (knowledge graph),
   `squashPatches` / `diffVersions` / `reconstructVersion` (version math),
   `AgentSession` (open/contribute/close lifecycle), and
   `ContributionReceipt` (audit envelope) behave per their documented
   invariants. CLEO trusts these for ADR-051 evidence chains and saga-level
   attestation.
3. **Failure mode predictability.** When the SDK is absent from the runtime,
   import sites throw `ERR_MODULE_NOT_FOUND` (Node ESM) or `MODULE_NOT_FOUND`
   (Node CJS). These are stable, catchable error codes — degraded-mode
   wrappers can rely on them.
4. **Library-shape exports.** Every type exported from any `llmtxt/*`
   subpath is a LIBRARY type — generated by and owned by the upstream
   project. The SDK's `AgentIdentity` class shape is publicly defined
   precisely because it is the cryptographic contract; CLEO trusts that
   shape and only that shape.

### §2.2 What CLEO MUST NOT assume

CLEO is forbidden from assuming:

1. **Application-level semantics.** The SDK has zero opinion on:
   - Slug uniqueness (`grep -rn 'allocateSlug\|reserveSlug' node_modules/
     llmtxt/dist/` returns 0 hits). `slug` is a scoping label, not an
     allocatable namespace — verified at `packages/cleo/.../node_modules/
     llmtxt/dist/blob/changeset.d.ts:64`: "The document slug to scope blob
     refs to (optional)". Slug uniqueness is **application-level — CLEO's
     responsibility to centralise** (Epic T10289).
   - DocKind taxonomy. `packages/contracts/src/docs-taxonomy.ts` defines
     `adr`, `spec`, `research`, `handoff`, `note`, `llm-readme`, `changeset`,
     `release-note`, `plan`, `rcasd`. The SDK treats every blob as opaque.
   - ADR numbering, similarity warnings, audit-log writes, idempotency
     fences. All of these are CLEO orchestration concerns.
2. **Stable shape for `optionalDependencies`.** Declaring `llmtxt` under
   `optionalDependencies` in `packages/core/package.json` affects ONLY
   install-time resolution — it does NOT prevent `ERR_MODULE_NOT_FOUND` at
   runtime when a static `import { X } from 'llmtxt/Y'` statement loads.
   CLEO MUST NOT assume static imports work when the SDK is absent; every
   import site that participates in degraded mode MUST be dynamic with
   `throwUnavailable` wrapping (see §2.3).
3. **SDK error shapes as public CLEO contracts.** When the SDK throws,
   CLEO MUST wrap the error into an LAFS envelope with a CLEO-owned error
   code (`LLMTXT_PRIMITIVE_UNAVAILABLE`, `LLMTXT_SIGNATURE_INVALID`,
   `LLMTXT_BLOB_NOT_FOUND`, etc.). Raw SDK errors MUST NOT cross CLEO's
   public CLI / dispatch boundary.
4. **Internal data structures.** Reaching into `llmtxt/internals/*` or
   relying on undocumented runtime shapes is forbidden. Only documented
   exports may be imported.

### §2.3 Peer-dep degraded-mode contract

`packages/core/package.json` declares `llmtxt` under
`optionalDependencies` AND `peerDependenciesMeta` with `optional: true`.
The T10354 audit found this declaration is internally inconsistent with
the codebase: 11 static `import { ... } from 'llmtxt/*'` statements crash
module-load if the SDK is absent, defeating the `optional` posture.

**This ADR REQUIRES** every existing and future SDK import site to follow
one of two patterns:

- **Pattern A (recommended for degraded-mode primitives):** Dynamic import
  inside a `throwUnavailable` wrapper. Canonical reference:
  `packages/core/src/docs/docs-ops.ts:43` — `await throwUnavailable(() =>
  import('llmtxt/identity'))`. The wrapper normalises
  `ERR_MODULE_NOT_FOUND` into `LLMTXT_PRIMITIVE_UNAVAILABLE`. Use this
  pattern for any code path that has a meaningful degraded behaviour
  (e.g. skip provenance, surface a warning, continue without the primitive).

- **Pattern B (recommended for required primitives):** Static import +
  promote `llmtxt` to hard `dependencies`. Use this pattern for code paths
  where the SDK is required (e.g. identity attestation in
  ADR-051 evidence chains). Document the requirement in the calling
  module's header JSDoc.

**Mixing the two without justification is a contract violation.** Sites
discovered by the T10354 audit at `packages/core/src/sentient/events.ts:503`
and `packages/core/src/sentient/state.ts:448` (F-4) use dynamic imports
WITHOUT the `throwUnavailable` wrapper — they MUST be migrated to
Pattern A by Epic T10292's remediation tasks.

A future CI gate (filed as a follow-up task under T10292 by this ADR's
acceptance criteria) will lint for raw `import('llmtxt/*')` calls that
are not wrapped in `throwUnavailable`. Until that lint lands, the
boundary is enforced by code review.

### §2.4 Type-leak prohibition

The T10354 / T10355 audits flagged three high-impact leaks:

- **F-1**: `AgentIdentity` re-exported through 4 public CLEO surfaces
  (`packages/core/src/identity/cleo-identity.ts:33`,
  `packages/core/src/sentient/events.ts:43`,
  `packages/core/src/sentient/revert-executor.ts:30`,
  `packages/core/src/sentient/kms.ts:30+346`).
- **F-2**: `DocsGraphResult.raw: KnowledgeGraph` embeds the SDK type
  directly in a public envelope at
  `packages/core/src/docs/docs-ops.ts:128-129`.
- **F-3**: `AgentSessionHandle.session: AgentSession` and
  `WrappedResult<T>.receipt: ContributionReceipt | null` leak SDK types
  through `packages/core/src/sessions/agent-session-adapter.ts:90-123`.

To prevent regression, this ADR establishes three prohibitions:

1. **No SDK types in `packages/contracts/src/**`.** Every new
   `export interface` or `export type` declared in any file under
   `packages/contracts/src/` MUST NOT reference any `llmtxt/*` symbol.
   If a contract needs to capture a primitive shape, define a CLEO-shaped
   interface (e.g. `CleoSigner` capturing the minimal sign/verify/pub
   surface CLEO actually uses) — do NOT re-export the SDK type.

2. **No SDK types in `packages/core/src/index.ts` (or any barrel file).**
   The core barrel is CLEO's public surface to other packages (`cleo`,
   `studio`, `caamp`, `cleo-os`, etc.). Re-exporting SDK types from the
   barrel forces every downstream consumer to build-time-depend on
   `llmtxt`, transitively defeating §2.3 Pattern A.

3. **No SDK types in result envelopes.** Any public envelope returned by
   a dispatch handler, CLI command, or core SDK function MUST contain only
   CLEO-owned types (defined in `packages/contracts/`). If the envelope
   needs to surface SDK-derived data (e.g. a knowledge graph), the
   contract MUST define a `KnowledgeGraphSerialized` (or equivalent)
   JSON-shaped CLEO contract and the envelope MUST contain THAT shape.

**Enforcement (deferred to follow-up):** a CI lint that scans
`packages/contracts/src/**` and `packages/core/src/index.ts` for the
substring `from 'llmtxt` will be filed as a follow-up subtask under
T10292. Until then, the prohibition is review-enforced.

**Migration path for existing leaks:**

- F-1 → define `CleoSigner` in `packages/contracts/src/identity/`;
  `cleo-identity.ts` adapts.
- F-2 → drop `DocsGraphResult.raw` (callers re-build via SDK directly)
  OR replace with `KnowledgeGraphSerialized` in contracts.
- F-3 → make `AgentSessionHandle.session` opaque (`unknown` +
  opaque-symbol tag); define `CleoContributionReceipt` in contracts.

These migrations are owned by Epic T10292's remediation children, not
this ADR.

### §2.5 Upstream issue escalation policy

When a future gap is discovered, classify the responsibility per the
T10355 matrix rules:

| Verdict | Definition | Action |
|---------|------------|--------|
| `CLEO` | Failure happens entirely inside CLEO's code (CLI shell, dispatch, contracts, persistence, orchestration). SDK is never invoked or is invoked correctly. | File as a child task under the responsible CLEO Epic (E1 slug-namespace, E2 writer-dedup, E3 CLI-hardening, E5 retro-normalize). NO upstream issue. |
| `shared-CLEO-led` | SDK is correctly providing a primitive, but CLEO's WRAPPER, RE-EXPORT, or ENVELOPE leaks SDK semantics into CLEO's public surface (F-1, F-2, F-3). The remediation is CLEO-side (reshape contracts, add adapters, drop barrel re-exports). | File as a child task under E4-DOCS-SDK-BOUNDARY (T10292). NO upstream issue — the SDK behaviour is correct. |
| `shared-SDK-led` | SDK API would need to change to resolve the gap (e.g. a missing primitive that CLEO would otherwise have to vendor). | File a GitHub Issue against `kryptobaseddev/llmtxt` with a complete reproduction + the proposed API. ALSO file a child task under T10292 tracking the upstream issue's progress. |
| `SDK` | SDK internal bug — primitive misbehaves vs its documented contract. | File a GitHub Issue against `kryptobaseddev/llmtxt`. CLEO-side work is limited to a temporary workaround until the upstream fix ships. |

**Current state per T10355 (frozen 2026-05-23):**

- 8 gaps verdict `CLEO` — handled by E1/E2/E3/E5 inside T10288.
- 2 gaps verdict `shared-CLEO-led` (F-1, F-2/F-3 cluster) — handled by
  T10292 itself.
- 0 gaps verdict `shared-SDK-led`.
- 0 gaps verdict `SDK`.

Therefore: **no upstream issues are filed by this ADR's acceptance.**
T10356 (E4.4 "File upstream issues") confirmed-empty per the verdict
matrix and was closed-as-NoOp.

If the classification policy itself changes (e.g. CLEO decides to file
upstream issues for "ergonomic" reasons even on `shared-CLEO-led`
verdicts), this ADR MUST be amended — not silently overridden by a
runbook elsewhere.

## §3 Consequences

### Positive

- **Clean ownership boundary.** Future gap classification has a
  reproducible algorithm. No more "should this be upstream or here"
  debates inside PR review.
- **Type-leak regression-locked.** The three prohibitions in §2.4 prevent
  the F-1/F-2/F-3 failure mode from re-occurring in new code paths.
- **Degraded-mode posture explicit.** The Pattern A / Pattern B choice in
  §2.3 ends the F-5 inconsistency between `optionalDependencies`
  declaration and 11 static imports — every future PR picks one pattern.
- **Upstream relationship documented.** `kryptobaseddev/llmtxt`
  maintainers (or anyone reading the boundary) can see exactly what CLEO
  consumes, what it does NOT consume, and what would trigger an upstream
  issue.
- **ADR-051 evidence chain robustness preserved.** The cryptographic
  attestation flow trusts SDK primitives explicitly (§2.1.2) — this ADR
  makes the trust boundary auditable.

### Negative

- **Migration cost for 2 shared-CLEO-led gaps.** F-1 (4 re-export sites)
  and F-2/F-3 (1 result envelope + 1 session adapter) require reshape
  work owned by T10292's remediation children. Scope is bounded: ~6 files
  total. Tracked as future subtasks under T10292.
- **Peer-dep migration cost.** Existing 11 static imports must be either
  converted to Pattern A or `llmtxt` must be promoted to hard
  `dependencies`. Choice is deferred to a follow-up subtask under T10292;
  this ADR REQUIRES the choice be made and documented.
- **CI lint gates are not in this PR.** The §2.3 dynamic-import lint and
  the §2.4 type-leak lint are deferred to follow-up subtasks. Until they
  ship, regressions are caught only by code review.

### Neutral

- This ADR is enforcement-by-policy, not enforcement-by-tooling, for its
  first cycle. The CI lints will be filed as follow-up subtasks under
  T10292; until they land, drift is review-enforced.
- The verdict matrix in T10355 is frozen at the 2026-05-23 snapshot.
  Future gaps will be classified using the §2.5 rules but will NOT
  retroactively re-classify the existing 10.
