## 1. Tier Definitions

### Mental Model

The three tiers map to the Letta OS-memory analogy applied to CLEO's project lifecycle:

```
SHORT-TERM  = RAM     ‚Äî session-scoped working context, volatile, fast
MEDIUM-TERM = Disk    ‚Äî project-scoped verified facts, weeks of retention
LONG-TERM   = Archive ‚Äî architectural bedrock, permanent, supersession-only eviction
```

### 1.1 Short-Term (Working Memory)

**Purpose**: Active session context ‚Äî what the agent is working on *right now*.

**What belongs here**:
- Session-scoped observations generated by lifecycle hooks (TaskStart, TaskComplete, AgentSpawn, WorkAdd) during the **current session**
- Sticky notes (status = 'active') ‚Äî already ephemeral by design
- Auto-extracted transcript learnings (confidence < 0.70, source = 'transcript:ses_xxx')
- Hook-generated observations (sourceType = 'agent', type = 'change' | 'feature') produced within the last 24 hours

**Storage indicator** (existing schema): `memoryTier = 'short'` on:
- `brain_observations` (new column)
- `brain_learnings` (new column)
- `brain_patterns` (new column)
- `brain_decisions` (new column)
- `brain_page_nodes.metadata_json` field `tier` (for graph layer; no separate column needed)

**Session relationship**: Short-term entries are linked to the session that produced them via `brain_observations.sourceSessionId` (already present). At session end, the consolidator promotes or evicts.

**Eviction rules**:
1. At `cleo session end` (after backup), the sleep-time consolidator runs.
2. Short-term observations from the **ending session** are scored:
   - qualityScore < 0.5 AND no task link ‚Üí **evict** (set `invalidAt = now()`, `memoryTier` unchanged so auditors can see eviction tier)
   - qualityScore >= 0.5 AND `verified = false` ‚Üí promote to **medium-term** pending verification
   - qualityScore >= 0.7 AND `verified = true` ‚Üí promote to **medium-term** directly
3. Sticky notes are **not** touched by the consolidator ‚Äî they have their own lifecycle (`active` ‚Üí `converted` ‚Üí `archived`)
4. Short-term entries older than 48 hours that have not been promoted are soft-evicted (set `invalidAt`)

**Token budget in bridge**: 0 tokens. Short-term entries do NOT appear in the static memory bridge (`.cleo/memory-bridge.md`). They are JIT-injected per-session via `getSessionMemoryContext()` which is already session-scoped.

---

### 1.2 Medium-Term (Project Memory)

**Purpose**: Verified project knowledge ‚Äî facts, decisions, and patterns that have been confirmed within the last 30 days.

**What belongs here**:
- Decisions with outcome = 'success' | 'pending' that have been in-use for < 30 days
- Learnings promoted from short-term after verification
- Patterns with frequency >= 2 (observed more than once)
- Observations manually entered (`sourceType = 'manual'`) that haven't been superseded
- Task-completion learnings generated by session-grade (confidence >= 0.70)

**Storage indicator**: `memoryTier = 'medium'`

**Promotion from short-term**: On session end, consolidator sets `memoryTier = 'medium'` based on quality + verification rules (see ¬ß1.1 eviction rules above).

**Promotion to long-term**: Consolidator checks medium-term entries on **every session end** (not just the producing session):
- Decision: has `outcome = 'success'` AND age > 7 days ‚Üí promote to long-term
- Learning: `confidence >= 0.80` AND age > 14 days AND `verified = true` ‚Üí promote
- Pattern: `frequency >= 5` AND `successRate >= 0.7` ‚Üí promote to long-term
- Observation: age > 30 days AND quality >= 0.7 AND `verified = true` ‚Üí promote

**Decay**: Medium-term learnings are subject to the existing `applyTemporalDecay()` (decayRate = 0.995, daily). Once effective confidence falls below 0.40 (not the existing 0.30 threshold ‚Äî raised for medium-tier), the entry is soft-evicted via `invalidAt`.

**Eviction**: Medium-term entries with `invalidAt IS NOT NULL` are excluded from all retrieval paths. They remain in the DB for audit (the `contradicts` edge and bitemporal query pattern require them).

**Token budget in bridge**: Up to 800 tokens from medium-tier content.

---

### 1.3 Long-Term (Permanent Memory)

**Purpose**: Architectural bedrock ‚Äî decisions, patterns, and facts that are proven, permanent, and define how the project works.

**What belongs here**:
- All ADR-backed decisions (type = 'architecture' | 'strategic', confidence = 'high')
- Procedural patterns (BRAIN patterns of type 'workflow' | 'optimization' with frequency >= 5 and successRate >= 0.70)
- Owner-stated facts (sourceType = 'manual' with quality >= 0.80)
- Session-debrief decisions (source_type = 'session-debrief')
- Community summaries (nodeType = 'summary' in brain_page_nodes)
- Proven learnings promoted from medium-term

**Storage indicator**: `memoryTier = 'long'`

**Eviction**: Long-term entries are NEVER automatically evicted. The only valid eviction paths are:
1. Explicit supersession: a newer decision creates a `supersedes` edge from new ‚Üí old, AND the old entry gets `invalidAt = now()` set by the consolidator.
2. Manual operator action via `cleo memory invalidate <id>` (new CLI op needed ‚Äî see ¬ß9).
3. Owner-stated facts can be explicitly retracted via `cleo memory retract <id> --reason "..."`.

**Enforcement**: The consolidator MUST skip long-term entries for automatic eviction. Query pattern:
```sql
WHERE memory_tier != 'long' AND invalid_at IS NULL
```

**Token budget in bridge**: Up to 1000 tokens from long-term content, always positioned first in the bridge (primacy bias ‚Äî see R5).

## 2. Memory Type Definitions

### Cognitive Taxonomy Mapped to Existing Schema

| Memory Type | Definition | Primary Tables | Distinguished By |
|-------------|-----------|----------------|-----------------|
| **Semantic** | Declarative facts ‚Äî "what is true about this project" | `brain_decisions`, `brain_learnings` | `memoryType = 'semantic'` |
| **Episodic** | Event records ‚Äî "what happened and when" | `brain_observations` | `memoryType = 'episodic'` |
| **Procedural** | Process knowledge ‚Äî "how to do things" | `brain_patterns` | `memoryType = 'procedural'` |

### 2.1 Semantic Memory (Facts)

**Maps to**: `brain_decisions` + `brain_learnings`

**Characteristics**:
- Timeless claims about codebase state, architecture, or owner preferences
- Examples: "The project uses Drizzle ORM v1.0.0-beta", "brain.db has 12 node types", "owner requires pipe-separated acceptance criteria"
- Confidence-rated; updated when new evidence contradicts
- Distinguished from episodic: semantic facts don't reference a specific event or timestamp ‚Äî they describe stable state

**Schema indicator**: `memoryType = 'semantic'`

**Default assignment at write time**:
- `brain_decisions` entries ‚Üí always `semantic`
- `brain_learnings` entries ‚Üí `semantic` by default; can be overridden to `episodic` if the insight is about a specific event

**Retrieval priority**: On-demand hybrid search. Included in bridge when `quality >= 0.7` AND `memoryTier IN ('medium', 'long')`.

### 2.2 Episodic Memory (Events)

**Maps to**: `brain_observations` (primary) + `brain_learnings` (event-specific insights)

**Characteristics**:
- Time-anchored event records ‚Äî something that *happened*
- Examples: "T523 was completed on 2026-04-12 ‚Äî purged 2,927 noise entries", "Session X shipped 3 releases", "Biome check failed due to import sorting at 2026-04-11"
- Tied to a specific session, task, or date
- Decays faster than semantic ‚Äî events become less relevant as time passes

**Schema indicator**: `memoryType = 'episodic'`

**Default assignment at write time**:
- `brain_observations` entries ‚Üí always `episodic`
- `brain_learnings` with `source` containing `transcript:ses_` ‚Üí `episodic`

**Bridge injection**: Episodic memories are **not injected into the static bridge** by default (too volatile, too session-specific). They are JIT-injected via `getSessionMemoryContext()` when a task/session context is active. Exception: session-debrief summaries (type = 'summary' in brain_page_nodes) are included in the bridge as consolidated episodic history.

**Decay**: Fastest-decaying type. The existing `applyTemporalDecay()` already targets `brain_learnings`. For episodic `brain_observations`: a read-time decay formula applies for bridge inclusion:
```
effectiveQuality = quality_score * 0.5 ^ (ageDays / 14)
```
Episodic entries with effective quality < 0.30 are excluded from bridge injection (not evicted from DB).

### 2.3 Procedural Memory (Processes)

**Maps to**: `brain_patterns`

**Characteristics**:
- How-to knowledge ‚Äî behavioral rules, workflow steps, coding standards
- Examples: "Always run `pnpm biome check --write .` before committing", "Use pipe `|` not multiple `--acceptance` flags", "Never use `git add -A` ‚Äî add specific files by name"
- High-frequency retrieval: injected into every session start unconditionally (top 5 by quality)
- Slower to form but slower to decay ‚Äî requires repeated evidence

**Schema indicator**: `memoryType = 'procedural'`

**Default assignment at write time**:
- `brain_patterns` entries ‚Üí always `procedural`

**Bridge injection**: Procedural memories are the **highest-priority bridge content**, always injected first. They occupy up to 500 tokens regardless of tier (short-term procedural patterns from a session are never injected ‚Äî only medium and long-tier).

**When promoted**: Pattern type `workflow` | `optimization` with frequency >= 5 ‚Üí promoted to long-term procedural.

**Eviction**: Long-term procedural entries are never auto-evicted. If a procedure is violated with consistently good outcomes (tracked via `brain_memory_links` and task completion patterns), the consolidator flags it for human review (does not auto-delete).

## 3. Schema Changes

### 3.1 New Columns Required

All four typed tables (`brain_decisions`, `brain_patterns`, `brain_learnings`, `brain_observations`) and the graph node table (`brain_page_nodes`) need the following additions.

#### 3.1.1 `memory_tier` Column

**Permitted path**: Extend existing locked tables via `ALTER TABLE ADD COLUMN` (R6 ¬ß7.1).

```typescript
// In brain-schema.ts ‚Äî add to all four typed tables + brain_page_nodes

export const BRAIN_MEMORY_TIERS = ['short', 'medium', 'long'] as const;
export type BrainMemoryTier = (typeof BRAIN_MEMORY_TIERS)[number];
```

**Drizzle column definition** (identical for all tables):
```typescript
memoryTier: text('memory_tier', { enum: BRAIN_MEMORY_TIERS }).default('short'),
```

**Default**: `'short'` ‚Äî every new entry starts as short-term and must be explicitly promoted. This is the "ground truth: do not store until verified" invariant ‚Äî entries must earn their way to medium/long.

**NULL semantics**: Legacy rows (pre-migration) get `NULL` for `memory_tier`. Query patterns MUST treat `NULL` as `'medium'` for backward compatibility (existing entries were high enough quality to survive T523 purge, so medium is a safe assumption).

#### 3.1.2 `memory_type` Column

```typescript

export type BrainMemoryTypeV2 = (typeof BRAIN_MEMORY_TYPES_V2)[number];

// NOTE: BRAIN_MEMORY_TYPES already exists in brain-schema.ts as the link table enum.

export const BRAIN_COGNITIVE_TYPES = ['semantic', 'episodic', 'procedural'] as const;
export type BrainCognitiveType = (typeof BRAIN_COGNITIVE_TYPES)[number];
```

**Drizzle column definition** (per-table default differs):
```typescript
// brain_decisions ‚Üí semantic
memoryType: text('memory_type', { enum: BRAIN_COGNITIVE_TYPES }).default('semantic'),

// brain_patterns ‚Üí procedural
memoryType: text('memory_type', { enum: BRAIN_COGNITIVE_TYPES }).default('procedural'),

// brain_learnings ‚Üí semantic (most learnings are factual; episodic learnings are exceptions)
memoryType: text('memory_type', { enum: BRAIN_COGNITIVE_TYPES }).default('semantic'),

// brain_observations ‚Üí episodic
memoryType: text('memory_type', { enum: BRAIN_COGNITIVE_TYPES }).default('episodic'),
```

**NULL semantics**: Legacy rows get `NULL`; treat as the table-default type at query time.

#### 3.1.3 `verified` Column

```typescript
// brain_decisions, brain_learnings, brain_observations (NOT brain_patterns ‚Äî patterns are verified by frequency)
verified: integer('verified', { mode: 'boolean' }).notNull().default(false),
```

**Semantics**:
- `false` (default): agent-inferred, unconfirmed, pending verification
- `true`: confirmed via one of the four verification gates (see ¬ß5)

**NULL semantics**: Not nullable ‚Äî `false` is the safe default.

**brain_patterns**: Uses `frequency` and `successRate` instead of a boolean verified flag. A pattern with `frequency >= 3` AND `successRate >= 0.6` is considered "verified" at query time without a column.

#### 3.1.4 `valid_at` and `invalid_at` Columns

```typescript
// brain_observations (primary bitemporal table)
// Also added to brain_learnings (temporal validity for facts that can change)
// NOT added to brain_decisions (decisions are superseded via graph edge, not timestamps)
// NOT added to brain_patterns (frequency/successRate handle this)

validAt: text('valid_at').notNull().default(sql`(datetime('now'))`),
invalidAt: text('invalid_at'), // NULL = currently valid
```

**Semantics**:
- `valid_at`: When this fact became true. Default = creation time. Can be backdated for historical facts.
- `invalid_at`: When this fact stopped being true. NULL = still valid. Set by the consolidator when a contradicting entry arrives.

**For `brain_decisions`**: Supersession is handled entirely by the `supersedes` edge type in `brain_page_edges` (already exists). No `invalid_at` column needed. The edge is the temporal signal. Querying active decisions: `WHERE NOT EXISTS (SELECT 1 FROM brain_page_edges WHERE from_id LIKE 'decision:%' AND to_id = 'decision:' || d.id AND edge_type = 'supersedes')`.

**For `brain_patterns`**: Lifecycle handled by `type` column ('blocker' vs 'success' vs 'failure') plus `frequency`/`successRate`. No `invalid_at` needed.

#### 3.1.5 `source_confidence` Column

This separates *source reliability* from *content quality* ‚Äî a critical distinction missing from the current schema (R1 ¬ß10, R2 ¬ß4.7):

```typescript
// All four typed tables + brain_page_nodes
export const BRAIN_SOURCE_CONFIDENCE = ['owner', 'task-outcome', 'agent', 'speculative'] as const;
export type BrainSourceConfidence = (typeof BRAIN_SOURCE_CONFIDENCE)[number];

sourceConfidence: text('source_confidence', { enum: BRAIN_SOURCE_CONFIDENCE }).default('agent'),
```

**Values and baseline quality multipliers** (used in updated quality scoring ‚Äî ¬ß5.3):
| Value | Meaning | Quality multiplier |
|-------|---------|-------------------|
| `owner` | Owner explicitly stated this fact | 1.0 (no penalty) |
| `task-outcome` | Verified by completed task with measurable result | 0.90 |
| `agent` | Agent-inferred during work (default) | 0.70 |
| `speculative` | Agent hypothesis, not yet corroborated | 0.40 |

**Assignment at write time** (routing logic):
- `cleo memory observe` with `--source owner` flag ‚Üí `owner`
- `session.grade` generated learnings ‚Üí `task-outcome`
- lifecycle hook auto-capture ‚Üí `agent`
- transcript auto-extract ‚Üí `speculative`
- `cleo memory decision-store` (manual) ‚Üí `owner`

## 4. Tier Routing Logic

### 4.1 Write-Time Tier Assignment

Every write path (observeBrain, storeDecision, storePattern, storeLearning) MUST assign `memoryTier`, `memoryType`, `sourceConfidence`, and `verified` at insert time. No deferred assignment.

#### Decision Tree for `memoryTier`

```
INPUT: { sourceType, sourceConfidence, confidence, manualFlag }

IF sourceType == 'manual' AND sourceConfidence == 'owner':
  ‚Üí start at 'medium' (owner-stated facts skip short-term)
  ‚Üí if confidence == 'high': start at 'long'

ELSE IF sourceType == 'session-debrief':
  ‚Üí start at 'medium' (debrief summaries are already synthesized)

ELSE IF (source contains 'transcript:ses_' OR sourceType == 'agent')
  AND confidence_score < 0.70:
  ‚Üí 'short' (transcript extracts and low-confidence agent observations)

ELSE IF sourceConfidence == 'task-outcome':
  ‚Üí 'medium' (task outcomes are verified by definition)

ELSE:
  ‚Üí 'short' (default ‚Äî consolidator promotes)
```

#### Decision Tree for `memoryType`

```
TABLE:            brain_decisions   ‚Üí 'semantic'   (always)
TABLE:            brain_patterns    ‚Üí 'procedural' (always)
TABLE:            brain_learnings   ‚Üí
  IF source contains 'transcript:ses_': 'episodic'
  ELSE: 'semantic'
TABLE:            brain_observations ‚Üí 'episodic'  (always)
```

#### Decision Tree for `sourceConfidence`

```
CALLER:           cleo memory decision-store (manual CLI)  ‚Üí 'owner'
CALLER:           cleo memory observe --source owner        ‚Üí 'owner'
CALLER:           session.grade (task outcome learnings)    ‚Üí 'task-outcome'
CALLER:           auto-extract transcript                   ‚Üí 'speculative'
CALLER:           lifecycle hook (task/agent/file/work)     ‚Üí 'agent'
CALLER:           cleo memory observe (no --source flag)    ‚Üí 'agent'
CALLER:           session-debrief                           ‚Üí 'task-outcome'
CALLER:           codebase-map store                        ‚Üí 'agent'
```

#### Decision Tree for `verified`

```
sourceConfidence == 'owner':         verified = true  (owner statement is ground truth)
sourceConfidence == 'task-outcome':  verified = true  (task completion is objective evidence)
sourceConfidence == 'agent':         verified = false (needs corroboration)
sourceConfidence == 'speculative':   verified = false (explicitly unverified)
```

### 4.2 Tier Promotion (Consolidator Logic)

The sleep-time consolidator runs at `cleo session end`, after the VACUUM INTO backup completes, as a fire-and-forget async process. It MUST NOT block session end.

```typescript
// Pseudocode ‚Äî consolidator promotion logic

async function runTierPromotion(db: BrainDb): Promise<void> {
  const now = new Date().toISOString();

  // 1. Promote short ‚Üí medium
  // brain_observations: quality >= 0.5, verified=true OR quality >= 0.7
  await db.update(brainObservations)
    .set({ memoryTier: 'medium' })
    .where(
      and(
        eq(brainObservations.memoryTier, 'short'),
        isNull(brainObservations.invalidAt),
        or(
          and(gte(brainObservations.qualityScore, 0.5), eq(brainObservations.verified, true)),
          gte(brainObservations.qualityScore, 0.7),
        ),
      ),
    );

  // 2. Evict short-tier entries older than 48h that weren't promoted
  const cutoff48h = new Date(Date.now() - 48 * 60 * 60 * 1000).toISOString();
  await db.update(brainObservations)
    .set({ invalidAt: now })
    .where(
      and(
        eq(brainObservations.memoryTier, 'short'),
        isNull(brainObservations.invalidAt),
        lt(brainObservations.createdAt, cutoff48h),
      ),
    );

  // 3. Promote medium ‚Üí long (decisions)
  const cutoff7d = new Date(Date.now() - 7 * 24 * 60 * 60 * 1000).toISOString();
  await db.update(brainDecisions)
    .set({ memoryTier: 'long' })
    .where(
      and(
        eq(brainDecisions.memoryTier, 'medium'),
        eq(brainDecisions.outcome, 'success'),
        lt(brainDecisions.createdAt, cutoff7d),
      ),
    );

  // 4. Promote medium ‚Üí long (patterns with high frequency + success rate)
  await db.update(brainPatterns)
    .set({ memoryTier: 'long' })
    .where(
      and(
        eq(brainPatterns.memoryTier, 'medium'),
        gte(brainPatterns.frequency, 5),
        gte(brainPatterns.successRate, 0.7),
      ),
    );

  // 5. Promote medium ‚Üí long (learnings: high confidence, verified, aged)
  const cutoff14d = new Date(Date.now() - 14 * 24 * 60 * 60 * 1000).toISOString();
  await db.update(brainLearnings)
    .set({ memoryTier: 'long' })
    .where(
      and(
        eq(brainLearnings.memoryTier, 'medium'),
        gte(brainLearnings.confidence, 0.8),
        eq(brainLearnings.verified, true),
        lt(brainLearnings.createdAt, cutoff14d),
      ),
    );
}
```

### 4.3 Can Tier Change Over Time?

| Direction | Allowed? | How |
|-----------|---------|-----|
| short ‚Üí medium | Yes | Consolidator promotion at session end |
| medium ‚Üí long | Yes | Consolidator after age + quality thresholds |
| long ‚Üí medium | No | Permanent by definition |
| long ‚Üí invalid | Yes (only) | Explicit supersession OR manual retract |
| medium ‚Üí invalid | Yes | Consolidator eviction + contradiction detection |
| short ‚Üí invalid | Yes | Consolidator 48h eviction |

## 5. Verification Gate Design

### 5.1 What Makes a Memory "Verified"

`verified = true` requires ONE of the following four gates to pass:

| Gate | Name | Condition | Who triggers |
|------|------|-----------|-------------|
| G1 | Owner Statement | sourceConfidence = 'owner' | Set at write time by CLI flag `--source owner` |
| G2 | Task Outcome | sourceConfidence = 'task-outcome' AND the linked task has status='completed' | Set at write time when source = session-grade or session-debrief |
| G3 | Corroboration | 2+ independent observations with content_hash similarity > 0.80 reach the same conclusion | Consolidator runs corroboration check at session end |
| G4 | Manual Confirmation | Operator explicitly runs `cleo memory verify <id>` | New CLI command (see ¬ß9) |

### 5.2 What Happens to Unverified Memories

Unverified (`verified = false`) entries follow a strict lifecycle:

1. **At creation**: assigned `memoryTier = 'short'` (unless G1/G2 qualify at write time)
2. **At first session end**: consolidator evaluates for promotion based on quality alone (G3 check runs)
3. **After 48 hours in short-tier without promotion**: `invalidAt` is set (soft eviction)
4. **After 30 days in medium-tier unverified**: `invalidAt` is set (medium-tier decay for unverified entries)
5. **In retrieval**: unverified entries are included but ranked lower ‚Äî quality multiplier of 0.7x applied

The existing quality_score system remains unchanged. `verified` is an ADDITIONAL gate, not a replacement. The compound gate: `quality_score >= 0.3 AND invalid_at IS NULL`.

### 5.3 Updated Quality Scoring Formula

Extend `computeObservationQuality()` and siblings in `quality-scoring.ts` to incorporate `sourceConfidence`:

```typescript
// Updated quality computation (all four tables):

const SOURCE_CONFIDENCE_MULTIPLIER: Record<BrainSourceConfidenceLevel, number> = {
  owner: 1.0,
  'task-outcome': 0.90,
  agent: 0.70,
  speculative: 0.40,
};

// Apply to existing score at the end of each compute function:
function applySourceMultiplier(
  rawScore: number,
  sourceConfidence: BrainSourceConfidenceLevel,
): number {
  return clamp(rawScore * SOURCE_CONFIDENCE_MULTIPLIER[sourceConfidence]);
}
```

This means an agent-inferred observation with raw score 0.65 becomes 0.65 √ó 0.70 = 0.455 effective. An owner-stated observation with raw score 0.65 stays at 0.65. Both pass the 0.30 threshold, but retrieval ranking separates them.

### 5.4 Verification's Effect on Retrieval Priority

In `searchBrain()` and `hybridSearch()`:

```sql
-- Add verified bonus to BM25 score in ORDER BY:
ORDER BY (
  bm25(brain_observations_fts) * -1
  * (1.0 + (verified * 0.2))  -- 20% boost for verified entries
  * quality_score
  * CASE WHEN memory_tier = 'long' THEN 1.3
         WHEN memory_tier = 'medium' THEN 1.0
         ELSE 0.7 END          -- tier score multiplier
) DESC
```

This gives long-term verified entries a strong advantage in retrieval ranking without hard-excluding unverified content.

## 6. Bitemporal Validity Design

### 6.1 Scope

Bitemporal validity applies to:
- `brain_observations` (primary ‚Äî events can be superseded by later events)
- `brain_learnings` (facts can change ‚Äî "owner uses X" might change to "owner uses Y")

**NOT applied to**:
- `brain_decisions`: handled by the `supersedes` edge in `brain_page_edges` (already in schema)
- `brain_patterns`: handled by `frequency` trending and `successRate`

### 6.2 Columns

- `valid_at TEXT` ‚Äî ISO 8601, defaults to `datetime('now')`. The moment this fact became true.
- `invalid_at TEXT` ‚Äî NULL until invalidated. The moment this fact stopped being true.

### 6.3 Contradiction Detection Pipeline

When a new observation/learning is written to brain.db, the consolidator (or an inline dedup check for `brain_learnings`) runs contradiction detection:

```typescript
async function detectContradictions(
  newEntry: { id: string; contentHash: string; narrative: string; memoryType: BrainCognitiveType },
  db: BrainDb,
): Promise<void> {
  if (newEntry.memoryType !== 'semantic') return; // Episodic entries don't contradict

  // Step 1: Find similar existing active entries via content hash prefix or embedding similarity
  const candidates = await db
    .select()
    .from(brainObservations)
    .where(
      and(
        isNull(brainObservations.invalidAt),
        ne(brainObservations.id, newEntry.id),
        eq(brainObservations.memoryType, 'semantic'),
      ),
    )
    .limit(20);

  // Step 2: For each candidate, check similarity (hash prefix match = duplicate; semantic = potential contradiction)
  for (const candidate of candidates) {
    const similarity = computeJaccardSimilarity(newEntry.narrative, candidate.narrative ?? '');

    if (similarity > 0.85) {
      // Near-duplicate: invalidate the older entry
      const older = newEntry.id < candidate.id ? newEntry : candidate; // ID ordering approximates age
      await db
        .update(brainObservations)
        .set({ invalidAt: new Date().toISOString() })
        .where(eq(brainObservations.id, older.id));

      // Add a graph edge: new ‚Üí contradicts ‚Üí old
      await addGraphEdge({
        fromId: `observation:${newEntry.id}`,
        toId: `observation:${candidate.id}`,
        edgeType: 'contradicts',
        weight: similarity,
        provenance: 'auto:contradiction-detected',
      });
    }
  }
}
```

**When contradiction detection runs**:
- For `brain_learnings`: inline within `storeLearning()` ‚Äî before insert
- For `brain_observations`: in the consolidator at session end (not inline ‚Äî too expensive for every hook event)

The inline dedup (SHA-256 30s window) already handles exact duplicates. Contradiction detection handles *semantic* contradictions where the content is different but the claim conflicts.

### 6.4 Graph Interaction

The existing `contradicts` edge type in `brain_page_edges` is the durable record of the contradiction relationship. The `invalid_at` column is the operational gate (filters out invalidated entries from retrieval). Together:

- **Read queries**: `WHERE invalid_at IS NULL` (active facts)
- **Audit queries**: `WHERE invalid_at IS NOT NULL` (historical/retracted facts)
- **Graph traversal**: `contradicts` edges lead to superseded facts for forensic reconstruction ("what did BRAIN believe before T549?")

### 6.5 Temporal Query Patterns

New query: "What did BRAIN know at a specific point in time?"

```sql
-- All active observations at a given timestamp
SELECT * FROM brain_observations
WHERE valid_at <= :targetTime
  AND (invalid_at IS NULL OR invalid_at > :targetTime)
ORDER BY quality_score DESC;
```

This enables the `cleo memory find --at "2026-04-11"` query planned in R2 ¬ß8.2.

### 6.6 Interaction with Existing `contradicts` Link Type in brain_memory_links

`brain_memory_links` has a `contradicts` link type in `BRAIN_LINK_TYPES`. This is a cross-table memory ‚Üí task link (memory entry contradicts a task claim). The new graph `contradicts` edge is memory ‚Üí memory. These are orthogonal ‚Äî no collision.

## Summary

This spec achieves the owner's requirements using ONLY the permitted extension path (new columns on existing tables, no new tables, no new ADR for the migration itself):

1. **Three tiers** defined with concrete Drizzle column (`memory_tier TEXT DEFAULT 'short'`) and deterministic promotion/eviction rules keyed to age + quality + verification
2. **Three cognitive types** mapped concretely to existing tables (decisions=semantic, patterns=procedural, observations=episodic) via `memory_type TEXT` column with per-table defaults
3. **Verification gate** with four gates: owner statement (write-time), task outcome (write-time), corroboration (consolidator), manual CLI (`cleo memory verify`)
4. **Bitemporal validity** via `valid_at` + `invalid_at` on observations and learnings; decisions use the existing `supersedes` graph edge
5. **Source confidence** as a separate dimension from content quality, driving a quality multiplier at scoring time
6. **Sleep-time consolidator** wired to session end, fire-and-forget, with tier promotion and contradiction detection
7. **Migration path** is additive-only: existing rows backfilled in the same migration SQL, no data loss, backward compatible reads
8. **Config-gated** behind `brain.tiering.enabled = false` default ‚Äî zero behavior change until opted in
