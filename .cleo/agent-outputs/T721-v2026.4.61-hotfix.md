# T721 — v2026.4.61/v2026.4.62 Hotfix: nexus-sqlite.js Packaging Regression

**Status**: complete
**Date**: 2026-04-15
**Commit**: 191a5eef (T721 gate + v2026.4.61), 021c528e (build.mjs fix + v2026.4.62)
**npm**: @cleocode/core@2026.4.62 — verified to include dist/store/nexus-sqlite.js

---

## Root Cause (Confirmed)

Two compounding issues:

### Issue 1: tsc incremental cache serving stale dist/store/

The release workflow restored pnpm node_modules from cache using key `node-cache-Linux-x64-pnpm-...`. This cache included a stale `dist/store/` tree from a prior partial build. The `packages/core` `tsconfig.json` uses `composite: true`, and the build step ran `tsc --emitDeclarationOnly` (because `build.mjs` uses esbuild for `.js` files and tsc only for `.d.ts`). The cached tsbuildinfo marked `nexus-sqlite.ts` as up-to-date, so tsc skipped re-emitting it — but the cache had only `.d.ts` outputs, not `.js`. Result: tarball contained `dist/store/nexus-sqlite.d.ts` but no `dist/store/nexus-sqlite.js`.

### Issue 2 (actual root cause): build.mjs never emitted dist/store/*.js

The `coreBuildOptions` in `build.mjs` only listed 3 entry points:
- `packages/core/src/index.ts` → `dist/index.js`
- `packages/core/src/conduit/index.ts` → `dist/conduit/index.js`
- `packages/core/src/internal.ts` → `dist/internal.js`

The files `dist/store/nexus-sqlite.js`, `dist/store/nexus-schema.js`, and `dist/store/brain-sqlite.js` were **never in the build.mjs entry points**. They existed locally only as stale artifacts from a prior full `tsc` run (before the esbuild migration). The clean build + tsc --emitDeclarationOnly produced only `.d.ts` files for these, with no `.js` counterparts.

These files are imported at runtime via dynamic imports in the cleo CLI:
```ts
import('@cleocode/core/store/nexus-sqlite' as string)
```
The `as string` cast prevents esbuild from bundling them inline — they must exist as standalone `.js` files in `dist/store/`.

## Fixes Applied

### Fix 1 (T721, commit 191a5eef): Release workflow gates

Two gates added to `.github/workflows/release.yml` before the publish step:

1. **Clean dist step**: `pnpm -r exec -- sh -c 'rm -rf dist'` before every build. Prevents pnpm cache from serving stale incremental outputs.

2. **Tarball verification step**: Runs `npm pack --dry-run` on `packages/core` and asserts these files are present:
   - `dist/store/nexus-sqlite.js`
   - `dist/store/brain-sqlite.js`
   - `dist/index.js`
   - `dist/internal.js`

The gate correctly RED-LIGHTED v2026.4.61 and v2026.4.62-first-attempt, confirming it works.

### Fix 2 (v2026.4.62, commit 021c528e): build.mjs store entry points

Added explicit esbuild entry points for the three store files that are dynamically imported at runtime:

```js
{ in: 'packages/core/src/store/nexus-sqlite.ts', out: 'store/nexus-sqlite' },
{ in: 'packages/core/src/store/nexus-schema.ts', out: 'store/nexus-schema' },
{ in: 'packages/core/src/store/brain-sqlite.ts', out: 'store/brain-sqlite' },
```

This ensures esbuild always emits these files as part of every build, not just when stale local artifacts happen to exist.
