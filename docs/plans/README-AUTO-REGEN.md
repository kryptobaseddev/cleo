# Plan: README Auto-Regen + forge-ts CI Gate Hardening

**Status**: Draft  
**Date**: 2026-04-20  
**Trigger**: Doc-sync audit found 8 packages with drift, forge-ts gate non-blocking  

---

## Problem

README files drift because they are maintained manually. The doc-sync audit found:

- 8 of 15 packages have documented drift (missing providers, stale status, missing subpaths)
- `packages/studio` has no README at all
- The root README package table is missing 3 packages (brain, playbooks, studio)
- forge-ts CI job exists but is non-blocking (`continue-on-error: true` + `|| true`)
- forge-ts cannot measure most packages (only `caamp` has a config)

---

## Proposal

Two parallel tracks:

### Track A — Harden forge-ts as a Real CI Gate

**Step 1: Fix root tsconfig incompatibility with forge-ts E009**

Add explicit flags to `tsconfig.json` (they are already implied by `"strict": true`
but forge-ts requires explicit presence):

```json
"strictNullChecks": true,
"noImplicitAny": true
```

**Step 2: Add per-package `forge-ts.config.ts` for priority packages**

Priority order (by public API surface and consumer count):

1. `packages/core` — ~4,534 exports, zero coverage today
2. `packages/contracts` — ~1,265 exports, the type SSoT
3. `packages/lafs` — publicly published, external consumers
4. `packages/adapters` — external consumer surface
5. `packages/runtime` — published package
6. `packages/nexus` — published package
7. `packages/brain` — first-class workspace package
8. `packages/playbooks` — internal but growing
9. `packages/cant` — napi binding surface
10. `packages/cleo` + `packages/cleo-os` — CLI entry points

Each config should mirror the `caamp` pattern: enforce `require-summary`,
`require-param`, `require-returns` as errors; `require-remarks` as warn;
release-tag and since rules off until audit lands.

**Step 3: Phase in CI gate enforcement**

| Release | Threshold | Action |
|---------|-----------|--------|
| v2026.5.x (next month) | 70% coverage per package | Remove `continue-on-error: true`; fail CI if `forge-ts check` exits non-zero in any configured package |
| v2026.6.x | 85% coverage per package | Bump threshold in config |
| v2026.7.x | 95% coverage per package | Target steady-state |

CI job change needed: remove `continue-on-error: true` and replace `|| true`
suffixes with hard fail. Add a matrix strategy to run forge-ts per configured
package rather than once at root.

---

### Track B — `pnpm run docs:readme-sync` Script

**Goal**: Keep the machine-generated sections of each README in sync with
package.json exports and forge-ts output. Human-authored narrative is preserved.

#### Mechanism

1. Each package README gets structured comment delimiters:

```markdown
<!-- auto:exports -->
<!-- /auto:exports -->
```

and in the root README:

```markdown
<!-- auto:package-list -->
<!-- /auto:package-list -->
```

2. A new script `scripts/readme-sync.mjs` runs after `pnpm run build`:

   a. Reads each `packages/*/package.json` and extracts `name`, `description`,
      `exports` keys, and the layer from the root README table (kept in a
      `packages/*/package-meta.json` sidecar).

   b. Generates an exports table for each package README between the
      `<!-- auto:exports -->` markers.

   c. Regenerates the root README `<!-- auto:package-list -->` block from
      all `packages/*/package.json` workspace members, sorted by layer then name.

3. forge-ts generates `docs/generated/api-reference.md` and `llms.txt`/`llms-full.txt`
   per configured package. The readme-sync script can pull the top-N exported symbols
   (those with TSDoc summaries) into a "Key Exports" section in each package README.

#### Root README structure after regen

```markdown
## Monorepo Structure

<!-- auto:package-list -->
<!-- This block is regenerated by `pnpm run docs:readme-sync` -->
<!-- Do not edit manually between these markers -->

| Layer | Package | Purpose |
|-------|---------|---------|
...all 15 packages...

<!-- /auto:package-list -->
```

#### Invocation

```bash
# One-shot regen
pnpm run docs:readme-sync

# Add to CI (informational, no fail)
pnpm run docs:readme-sync --check   # diff only, non-zero exit if drift
```

The `--check` mode can be added to CI as a soft gate initially, becoming
hard-fail once the initial sync is complete.

---

## What This Does NOT Do

- Does not auto-rewrite narrative prose (founding story, "why" sections)
- Does not auto-generate per-function docs in READMEs (use forge-ts build for that)
- Does not enforce consistent style across human-authored sections

---

## Implementation Order

1. Fix forge-ts E009 root tsconfig issue (1 line change, trivial)
2. Add forge-ts configs for `core` and `contracts` (highest ROI)
3. Remove `continue-on-error: true` from CI for configured packages
4. Build `scripts/readme-sync.mjs` for the auto block generation
5. Phase in remaining package configs over the next 2 releases
6. Enable `--check` mode in CI once initial sync run passes
