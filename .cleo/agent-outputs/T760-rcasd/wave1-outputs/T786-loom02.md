## Summary

`cleo add --type epic --lifecycle auto` now bootstraps the RCASD lifecycle
pipeline at the `research` stage immediately after epic creation. Default
behavior (`--lifecycle off`) is fully preserved — no change to existing callers.
The implementation is idempotent and fault-tolerant: if lifecycle init fails the
epic is still created and a warning is written to stderr.

## Files Modified (4)

### 1. `packages/cleo/src/dispatch/engines/lifecycle-engine.ts`

Added `lifecycleAutoInit(epicId, projectRoot?)` — a safe bootstrap function that:
- Calls `getLifecycleStatus` to check if the pipeline already exists
- If not initialized: calls `recordStageProgress(epicId, 'research', 'in_progress')` directly
  (bypasses `checkGate` and forward-only enforcement — same rationale as T785)
- Returns `{ lifecycleInitialized: boolean, stage: string }`
- Idempotent: second call on an already-initialized epic returns `lifecycleInitialized: false`

### 2. `packages/cleo/src/dispatch/lib/engine.ts`

Added `lifecycleAutoInit` to the lifecycle engine re-export block so downstream
consumers can access it without importing the engine directly.

### 3. `packages/cleo/src/cli/commands/add.ts`

- Added `--lifecycle <mode>` option (default: `off`) with help text:
  "Auto-initialize LOOM pipeline at research stage when epic is created (auto|off)"
- After epic creation, when `--lifecycle auto` AND `--type epic` AND not dry-run:
  - Calls `lifecycleAutoInit(epicId)`
  - On success: merges `{ lifecycleInitialized, stage }` into the output data
  - On failure: writes a `⚠` warning to stderr; the epic is never rolled back
- Default `off` path: zero code change — no lifecycle call, no output difference

### 4. `packages/cleo/src/dispatch/engines/__tests__/lifecycle-engine.test.ts`

## Design Notes

- **Safe init path**: `recordStageProgress` used directly (same as T785's
  orchestrate-engine) rather than `lifecycleProgress`, which calls `checkGate`
  and would block in strict enforcement mode on a fresh epic with no prior data.
- **Guardrail preserved**: `--lifecycle off` (default) path has zero code change
  — the action handler behaves identically to before T786.
- **Fault-tolerant**: lifecycle failure never aborts the `add` operation. The
  epic is created first, lifecycle bootstrap is best-effort.
- **No new dispatch operations**: the flag is CLI-only, handled entirely in the
  action handler after the dispatch call resolves.
- **Accepted test limitation**: after `recordStageProgress('research', 'in_progress')`,
  `getLifecycleStatus` returns `initialized: true` but `currentStage: null`
  (because `currentStage` is only set on `completed`/`skipped` stages per core
  logic). Tests assert on the `stages` array entry instead.
