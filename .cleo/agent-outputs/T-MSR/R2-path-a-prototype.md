## 1. What Path A Would Delete

Path A deletes the entire `drizzle/` directory.

## 2. What Path A Would Preserve

### 2.1 drizzle-orm runtime dependency

`drizzle-orm` (`1.0.0-beta.22-ec7b61d`) stays. It provides:
- `NodeSQLiteDatabase` ‚Äî the typed query builder used throughout `packages/core/src/`
- `readMigrationFiles` / `migrate` ‚Äî used by `migration-manager.ts`
- Table schema definitions (`sqliteTable`, column helpers) ‚Äî used in all 5 schema files

Path A only removes drizzle-**kit** (the CLI/codegen tool). The ORM runtime is
a distinct package and is not affected.

### 2.2 Custom reconciler (migration-manager.ts) becomes SSoT

`packages/core/src/store/migration-manager.ts` already IS the authoritative
migration runner. It handles:
- Journal bootstrapping (Scenario 1)
- Orphaned hash reconciliation (Scenario 2 sub-cases A and B)
- Partially-applied migration detection and completion (Scenario 3, T920)
- Drizzle v1 beta name-backfill (Scenario 4)
- SQLITE_BUSY retry with exponential backoff
- Sanitization of whitespace-only statements from trailing `-->
  statement-breakpoint` markers

Under Path A, `migration-manager.ts` continues unchanged. It delegates to
`drizzle-orm/node-sqlite/migrator` for execution, which stays. The only
behavioral change is that new migrations are hand-authored SQL rather than
generated by drizzle-kit. The reconciler needs no modification for Path A.

### 2.3 Signaldock migration runner

`packages/core/src/store/signaldock-sqlite.ts` has its own embedded migration
runner (`_signaldock_migrations` table, `applyGlobalSignaldockSchema`). It
already does not use drizzle-kit. Path A has zero impact on signaldock.

### 2.4 All canonical migration.sql files

All 34 migration folders in `packages/core/migrations/` are preserved unchanged.
Path A only removes the snapshot.json files nested within them. The SQL DDL is
the source of truth; the snapshots are only used by drizzle-kit's diff engine.

## 3. What Path A Loses (drizzle-kit features with real consumers)

### 3.1 Partial index schema-level expression (`.where()` on IndexBuilder)

**What it is**: drizzle-kit's schema DSL would, in principle, allow defining
partial indexes via `.where()` on an `IndexBuilder`. This is not currently
supported by drizzle-orm beta.22's `sqliteTable` callback.

**Current state**: Already non-functional. The T1126 partial index
(`idx_tasks_sentient_proposals_today`) was authored as raw SQL in
`20260421000001_t1126-sentient-proposal-index/migration.sql` with an explicit
comment:
```
-- Drizzle ORM schema note: partial indexes with .where() are not yet
-- supported in the sqliteTable callback style used by tasks-schema.ts.
-- This migration is the canonical source of truth for the index.
```

**Conclusion**: Path A loses nothing here ‚Äî this feature was already unusable
and the team already adopted the hand-rolled workaround as the canonical
approach. No current task or documented workflow depends on drizzle-kit
generating partial indexes.

### 3.2 drizzle-kit studio

**What it is**: A browser-based GUI for browsing and editing database contents,
served locally via `drizzle-kit studio`.

**Current consumers**: The `db:studio`, `db:studio:brain`, `db:studio:nexus`
scripts in `package.json`. No documented workflow or task acceptance criterion
references studio as a required tool. No ADR or RCASD plan mentions studio as
part of a production workflow. Usage appears to have been ad-hoc developer
convenience.

**Alternative**: `sqlite3` CLI or any SQLite browser (DB Browser for SQLite,
TablePlus). The `cleo admin` commands also surface DB state.

**Conclusion**: Low-risk loss. No task or documented workflow has a hard
dependency on drizzle-kit studio.

### 3.3 drizzle-kit push

**What it is**: `drizzle-kit push` directly applies the current schema to the
database without generating a migration file. Not used in this project ‚Äî there
are no `db:push` scripts in `package.json`. The project uses the migration-based
workflow exclusively.

**Conclusion**: Not used. Zero impact from removal.

### 3.4 Diff-against-snapshot workflow

**What it is**: Running `drizzle-kit generate` compares the current TypeScript
schema against the last `snapshot.json` and emits only the delta as a new
migration. This makes adding a column to `tasks-schema.ts` and running
`pnpm db:generate` produce a clean `ALTER TABLE` migration automatically.

**Current state**: The workflow is already broken by the scratchpad divergence.
The scratchpad's `snapshot.json` files do not match the canonical tree state.
Running `drizzle-kit generate` against the current schema produces migrations
with auto-generated slugs (`complex_vampiro`, `melted_wind_dancer`) rather than
task-linked names. The canonical tree has 15 task folders in `drizzle-tasks/`
while the scratchpad has 6. Any diff generated today would be against a stale
snapshot.

**Conclusion**: The diff-against-snapshot workflow is functionally broken
today. Removing it formalizes what is already true.

## 4. What Path A Gains

- **Deterministic authoring**: Migration filenames are authored by the developer,
  not auto-generated. Names like `20260421000002_t1126-sentient-proposal-index`
  are always task-linked, reviewable, and auditable.
- **No beta-kit exposure**: drizzle-kit is a beta package that has caused
  timestamp collisions (see linter output: `20260421000001` appears twice in
  `drizzle-tasks`), auto-generated nonsense names, and a diverged scratchpad.
  Removing it eliminates a class of footgun.
- **No snapshot-chain maintenance**: The 11 snapshot.json files in the canonical
  tree are maintenance weight that currently diverges silently from the schema
  TS files. Hand-rolling removes the need to keep them synchronized.
- **Linter + runtime guards cover the footguns**: `scripts/lint-migrations.mjs`
  (this prototype) catches trailing breakpoints, timestamp collisions, orphan
  snapshots, and bad folder names. The existing `sanitizeMigrationStatements()`
  in `migration-manager.ts` guards against runtime breakpoint crashes. Together
  they replace the negative value that drizzle-kit's codegen was meant to
  provide.
- **Playbooks test already handles hand-authored SQL**: The `applyMigration`
  helper in `packages/playbooks/src/__tests__/starter.e2e.test.ts` splits on
  `statement-breakpoint` itself ‚Äî it already handles hand-authored migrations
  that include or omit the marker.

## 6. Hand-roll Workflow Draft (README.md Skeleton)

The following is a draft skeleton for `packages/core/migrations/README.md`:

---

```markdown
# CLEO Migration Authoring Guide

## DB Sets

| Directory | DB File | Migration Runner |
|-----------|---------|-----------------|
| `drizzle-tasks/` | `tasks.db` (project) | `migration-manager.ts` ‚Üí drizzle-orm |
| `drizzle-brain/` | `brain.db` (project) | `migration-manager.ts` ‚Üí drizzle-orm |
| `drizzle-nexus/` | `nexus.db` (global) | `migration-manager.ts` ‚Üí drizzle-orm |
| `drizzle-telemetry/` | `telemetry.db` (project) | `migration-manager.ts` ‚Üí drizzle-orm |
| `drizzle-signaldock/` | `signaldock.db` (global) | `signaldock-sqlite.ts` (embedded runner) |

## Naming Convention

Each migration lives in a folder: `packages/core/migrations/<db-set>/<timestamp>_<slug>/migration.sql`

- `<timestamp>`: 14-digit UTC timestamp: `YYYYMMDDHHmmss`
- `<slug>`: lowercase alphanumeric slug with hyphens, task-linked (e.g., `t1153-path-a`)
- Full folder name must match: `^\d{14}_[a-z0-9-]+$`
- Timestamp must be unique within the DB set ‚Äî check existing folders before creating

Generate a safe timestamp:
```bash
date -u +"%Y%m%d%H%M%S"
```

## Authoring Steps

1. Pick the correct DB set for your schema change.
2. Generate a timestamp: `date -u +"%Y%m%d%H%M%S"`
3. Create the folder: `mkdir packages/core/migrations/<db-set>/<timestamp>_<slug>`
4. Create `migration.sql` with your DDL.
5. Use `-->  statement-breakpoint` on its own line to separate statements if
   needed ‚Äî but prefer single-statement migrations. Do NOT end the file with
   `-->  statement-breakpoint`.
6. Run the linter: `node scripts/lint-migrations.mjs`
7. Test locally: `pnpm run test` exercises the migration runner on an in-memory DB.
8. For partial indexes or other features not expressible in schema TS, the raw
   SQL migration IS the canonical definition. Add a comment to the schema TS
   file pointing to the migration folder.

## Reconciler Edge Cases

`migration-manager.ts:reconcileJournal()` handles four recovery scenarios.
When adding a new scenario (e.g., a new type of DDL the probe doesn't understand):
1. Add a new `if` branch in `probeAndMarkApplied()` for the DDL pattern.
2. Write a unit test in `packages/core/src/store/__tests__/migration-manager.test.ts`.
3. Document the scenario number and trigger condition in the function JSDoc.

## Coordinating Across DB Sets

- Each DB set is independent ‚Äî migrations in `drizzle-tasks/` do not affect `drizzle-brain/`.
- If a schema change touches multiple DB sets (e.g., adding a cross-DB join column),
  create one migration per DB set with the same timestamp prefix and coordinated slugs.
- The timestamp collision linter (`RULE-2`) will catch accidental same-timestamp conflicts
  within a DB set. Cross-DB-set collisions are not violations.

## Signaldock Migrations

`drizzle-signaldock/` is handled by a separate embedded runner in
`signaldock-sqlite.ts`. Follow its embedded `MIGRATION_STEPS` array pattern
for new signaldock migrations ‚Äî do NOT add them to `drizzle-signaldock/`
as folder-based migrations.
```

## 7. Reconciler Continuity ‚Äî Assumptions That Break Under Path A

Under Path A, `migration-manager.ts` calls `readMigrationFiles({ migrationsFolder })`
from `drizzle-orm/migrator`. This function reads migration folders and parses
`migration.sql` ‚Äî it does NOT read `snapshot.json`. The snapshots are only
consumed by drizzle-kit's diff engine, not by the runtime migrator.

**No assumptions break.** The reconciler's four scenarios (bootstrap, orphan
hashes, partially-applied, null-name backfill) all operate on:
- `migration.sql` file content (SQL statements)
- `__drizzle_migrations` journal table (hash, name, created_at)
- SQLite `PRAGMA table_info()` and `sqlite_master` queries

None of these depend on `snapshot.json` existence.

**One assumption to verify**: `readMigrationFiles` computes a hash of the
migration file content (SHA256 or similar). Hand-authored SQL that contains
trailing whitespace, CRLF line endings, or UTF-8 BOM could produce hash
mismatches between environments. The existing `sanitizeMigrationStatements()`
function already guards against whitespace-only statement chunks. A convention
of LF-only line endings and no BOM (enforced by `.editorconfig`) closes this
risk.

The single RULE-2 ERROR is a real bug: the timestamp collision means migration
ordering is non-deterministic for those two `drizzle-tasks` migrations. The
reconciler processes them in filesystem readdir order, which is not guaranteed.
This bug exists independently of Path A and should be fixed regardless.

The 24 RULE-3 warnings are the expected consequence of the team abandoning
drizzle-kit generation partway through ‚Äî most DB sets have initial snapshots
but hand-authored later migrations without snapshots. Under Path A, all
snapshots would be deleted, making the warning count drop to 0 for RULE-3
(since the rule would never trigger without any snapshots present).

## 9. Decision Inputs for RCASD

This research is neutral evidence. The data points that most matter:

1. **The diff-against-snapshot workflow is already broken** ‚Äî not a future loss,
   a current non-function. The scratchpad is so diverged from the canonical tree
   that `drizzle-kit generate` cannot produce correct output today.

2. **The timestamp collision** (`20260421000001` √ó 2) was caused by
   drizzle-kit-adjacent workflow (two agents writing migrations in the same
   second). The linter detects it; fixing it is a one-folder rename.

3. **The reconciler has zero code changes** under Path A ‚Äî it already works
   without snapshots. The only code change Path A requires is removing drizzle-kit
   from `package.json` and deleting files.

4. **Path B** (keep drizzle-kit, fix snapshot chain) would require regenerating
   all 11 missing snapshots, resolving the scratchpad divergence, and imposing a
   discipline that has already failed twice (the scratchpad exists precisely
   because the discipline slipped).
