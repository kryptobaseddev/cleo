# CleoOS: The Agentic Development Environment

**Version**: 2026.4.8 — aligned with shipped reality
**Canonical Plan**: `docs/plans/CLEO-ULTRAPLAN.md`
**Date**: 2026-04-08

---

## 1. What CleoOS Is

CleoOS is the full Agentic Development Environment built on top of `@cleocode/core`.

Where `@cleocode/core` is the kernel -- tasks, sessions, memory, orchestration, lifecycle -- CleoOS is the complete operating system: a wrapped Pi harness, a CANT runtime bridge, JIT agents, and the intelligence fabric that turns a solo developer and their AI agents into a governed, continuous software development operation.

CleoOS wraps Pi (`@mariozechner/pi-coding-agent`) as its execution host. The `cleoos` launcher is an ~86-line wrapper that imports `AgentSession` from Pi, registers the CANT runtime bridge as an extension, and delegates the event loop to Pi. Zero fork burden. Pi stays upstream. `cleo` still works. `pi` still works. `cleoos` is the batteries-included binary.

> One developer. Many agents. One operating system for the work.

The kernel provides the primitives: create a task, start a session, observe a fact, check a gate, query the brain. CleoOS provides the higher-order behaviors: JIT agent composition with context-aware spawning, multi-agent coordination through the 3-tier hierarchy, model routing by task complexity, mental model reinforcement loops, and worktree-isolated parallel execution.

This boundary is not decorative. It is the same separation that exists between a kernel and an operating system in any serious system design. The kernel is stable, tested, and independently consumable. The OS is opinionated, adaptive, and built for a specific workflow: solo developer + AI agents + real software projects.

## 3. Why CleoOS

The problem is no longer "can AI write code." The problem is "can AI sustain a software project."

Writing code is a burst operation. Sustaining a project is a campaign: weeks of context, evolving requirements, accumulated decisions, shifting priorities, multiple agents that must coordinate without drift, and a developer who needs to trust the system to remember what happened yesterday and act on it tomorrow.

Current tools solve pieces of this:

- Code generation tools write functions
- Task managers track tickets
- Memory systems store observations
- Orchestration frameworks spawn agents

None of them solve the whole problem. None of them provide an integrated environment where tasks, memory, lifecycle, orchestration, and multi-agent coordination operate as a single coherent system with provenance, validation, and continuity.

CleoOS solves the whole problem by assembling the four canonical systems into a unified development environment:

| System | Kernel Role | CleoOS Extension |
|--------|------------|------------------|
| **BRAIN** | Store observations, patterns, learnings, decisions | Active memory circulation, contradiction detection, knowledge graph, temporal reasoning, semantic retrieval |
| **LOOM** | Lifecycle gates, stage transitions, pipeline stages | Autonomous progression through RCASD-IVTR+C, typed CANT protocol definitions, JIT agent dispatch per stage |
| **NEXUS** | Cross-project registry, federated queries | Organization-scale coordination, global pattern libraries, shared intelligence |
| **LAFS** | Structured envelopes, exit codes, progressive disclosure | Provider-neutral runtime contract for all agent communication |

## 4. Key Components

### 4.1 Autonomous Runtime

The autonomous runtime is the execution layer that makes CLEO self-propelling. CleoOS does NOT ship background daemons. The "daemon" behaviors described in the v1 draft are realized as **Pi extension lifecycle hooks**:

| Concern | Realization |
|---------|-------------|
| **Agent watchdogs** | `cleo-cant-bridge.ts` hooks on `before_agent_start` and `agent_end` |
| **Envelope parsing** | Bridge's LAFS parser on subagent stdout streams (via `PiHarness.spawnSubagent`) |
| **Self-propelling execution** | Pi's interactive session loop + bridge's JIT composer |
| **HITL rules** | Typed `constraints [approval]:` blocks in `.cant` files |
| **Quality patrol** | Validation-lead dispatch via 3-tier hierarchy (no separate patrol engine) |
| **Decomposition** | CANT `kind: lifecycle` definitions + JIT agent spawn per RCASD stage |

Pi's `AgentSession` owns the event loop. CleoOS owns the behavior layered on top. **No extra processes.**

### 4.2 Conduit Protocol

Conduit is the live agent-to-agent coordination path within CleoOS, realized as the **chat room extension** (`cleo-chatroom.ts`, 371 LOC).

There is no Rust broker, no dead-letter queue, no separate transport. The chat room uses Pi's existing `spawnSubagent` for agent creation, child session JSONL files for message history, and `pi.ui.setWidget()` for the TUI panel. Four tools (`send_to_lead`, `broadcast_to_team`, `report_to_orchestrator`, `query_peer`) provide structured inter-agent communication. See Section 4.13 for the full tool contract.

### 4.3 Provider Ecosystem

CleoOS is provider-neutral by design. The adapter system in `@cleocode/adapters` provides a bridge to every major AI coding tool:

- **Pi** -- First-class harness via `@cleocode/cleo-os`. Pi is the execution host for CleoOS; the `PiHarness` class in `packages/caamp/src/core/harness/pi.ts` provides `spawnSubagent`, `configureModels`, session management, theme/prompt installation, and CANT verb support (`pi cant install/remove/list`).
- **Claude Code** -- Full adapter (14 hooks + spawn + install) -- upgraded via CAAMP normalizer
- **OpenCode** -- Full adapter (10 hooks + spawn + install)
- **Cursor** -- Full adapter (10 hooks + install)
- **Gemini CLI** -- Full adapter (10 hooks + getTranscript + install)
- **Codex** -- Partial adapter (3 hooks + getTranscript + install)
- **Kimi** -- Minimal adapter (install only)
- **Future providers** -- Any tool that implements the `CLEOProviderAdapter` interface (or shells out to `cleo <command>`)

Discovery is manifest-based: each adapter declares detection patterns (environment variables, files, CLI availability), and the `AdapterManager` in core automatically activates the matching adapter at startup.

The Memory Bridge ensures every provider starts with context: a static seed file (`.cleo/memory-bridge.md`), guided self-retrieval via the `ct-memory` skill, and on-demand lookup via `cleo memory find`, `cleo memory timeline`, and `cleo memory fetch`.

### 4.4 Project Lifecycle

CleoOS manages the full arc of a software project through the RCASD-IVTR+C lifecycle, with each stage mapped to a JIT agent dispatch:

| Phase | What CleoOS Does | JIT Agent Chain |
|-------|-----------------|-----------------|
| **Research & Consensus** | Gather information, validate decisions | cleo-prime -> planning-lead -> product-manager + ux-researcher |
| **Architecture** | ADRs captured in tasks.db, linked to brain observations | planning-lead -> product-manager |
| **Decomposition** | Task breakdown + wave planning | planning-lead -> orchestrator |
| **Implementation** | Multi-agent waves in worktree isolation | engineering-lead -> frontend-dev + backend-dev |
| **Validation** | Four-layer anti-hallucination checks, gate enforcement | validation-lead -> qa-engineer + security-reviewer |
| **Release** | Version bump, changelog generation, ship pipeline | cleo-prime consensus with all leads |
| **Contribution** | Cross-cutting provenance and artifact publish | All leads |

Every phase is governed by LOOM's RCASD-IVTR+C pipeline with explicit gate transitions. No phase is skipped silently. Each stage has a typed `.cant` protocol definition (12 protocol files shipped in `packages/core/src/validation/protocols/cant/`).

### 4.5 Brain Intelligence

The BRAIN system in `@cleocode/core` stores observations, patterns, learnings, and decisions. CleoOS extends this into active intelligence:

- **Three-layer retrieval** (shipped): search/timeline/fetch with ~10x token savings over traditional RAG
- **FTS5 full-text search** (shipped): keyword search across all brain tables
- **Vector similarity** (shipped): Local embedding via `@xenova/transformers` all-MiniLM-L6-v2 -- dynamic import, zero overhead when disabled
- **Session summarization** (shipped): Dual-mode prompt + structured `SessionSummaryInput` ingestion on `session.end`
- **Memory bridge automation** (shipped): Hook-driven refresh on `session.end` and `tasks.complete` with 30-second debounce
- **Brain maintenance** (shipped): `cleo brain maintenance` combines temporal decay + consolidation + embedding backfill
- **Mental models** (shipped, Wave 8): Per-agent persistent mental models in `brain.db:agents/<name>/model` scoped per `projectHash`. Async-write via BRAIN observation queue, validate-on-load per L5. See Section 4.12.
- **Knowledge graph** (planned): relationship-based discovery with temporal reasoning
- **Active circulation** (planned): Reinforcement, contradiction detection, proactive context surfacing

The goal is a brain that does not merely store what happened, but actively improves the next action.

### 4.6 Nexus Network

NEXUS connects isolated CLEO projects into a federated intelligence network:

- **Project registry** -- each project is identified by a unique `projectHash` and registered in the global `~/.local/share/cleo/nexus.db`
- **Cross-project dependencies** -- `project:task_id` syntax enables dependencies that span project boundaries
- **Federated queries** -- search tasks, patterns, and learnings across all registered projects
- **Permission model** -- three-tier access (read/write/execute) preserves project sovereignty
- **Global pattern library** -- patterns and learnings from one project benefit all registered projects

At organizational scale, NEXUS transforms isolated project brains into a collective intelligence that learns across the entire development portfolio.

> **Phase 3 Deferral (T045 Assessment)**: A production usage audit conducted 2026-03-22 found that zero Nexus operations have been invoked outside of automated tests in 15+ days of availability. All 22 registered operations are implemented and tested (~5,753 production LOC, ~4,423 test LOC), but no real workflow has exercised cross-project coordination. Nexus is formally deferred to Phase 3.

### 4.7 Pi Harness Wrapping

CleoOS wraps Pi rather than forking it (locked decision L1). The integration surface:

- **`@cleocode/cleo-os`** (`packages/cleo-os/`): the `cleoos` binary -- an ~86-line launcher that imports `AgentSession` from `@mariozechner/pi-coding-agent`, registers the CANT bridge and chat room as `additionalExtensionPaths`, and calls `session.run()`.
- **`PiHarness`** (`packages/caamp/src/core/harness/pi.ts`): the existing CAAMP harness providing `spawnSubagent`, `configureModels`, session JSONL management, theme/prompt installation, and the `pi cant install/remove/list` verbs.
- **XDG path resolution** (`packages/cleo-os/src/xdg.ts`): resolves `~/.local/share/cleo/` hub, extension paths, and auth storage.

Three binaries, three audiences, one shared core: `cleoos` (batteries-included), `cleo` (raw CLI), `pi` (raw Pi).

### 4.8 CANT Runtime Bridge

The CANT runtime bridge (`cleo-cant-bridge.ts`) is the **single Pi extension** that connects typed CANT source to Pi's execution model:

- **`compileBundle()`** at `session_start`: loads and compiles all `.cant` files from the project's CANT tree into an in-memory bundle.
- **Append-only injection**: on `before_agent_start`, the bridge APPENDS to Pi's system prompt (per L6 -- never replaces). The injected content includes the agent's compiled CANT definition, resolved context sources, and validated mental model.
- **LAFS envelope parsing**: on subagent stdout streams, the bridge evaluates `success` fields against CANT decision trees to determine next state.
- **Error retry**: on LAFS error envelopes, CANT rules dictate retry logic programmatically.

The bridge replaces every hand-written pi-ext extension with typed CANT source compiled and dispatched at runtime.

### 4.9 JIT Agent Composition

JIT agents are composed at spawn time from three sources:

1. **Static CANT source** -- the agent's `.cant` file (prompt, constraints, tools, permissions, role, tier)
2. **Dynamic context** -- pulled from BRAIN at spawn time via `context_sources:` declarations
3. **Mental model** -- per-project persistent observations the agent has accumulated

Token budgets are enforced per tier at compose time:

| Tier | System prompt cap | Mental model cap | Context sources cap |
|------|-------------------|------------------|---------------------|
| low  | 4k                | 0                | 0 (prompt only)     |
| mid  | 12k               | 1k               | 4k                  |
| high | 32k               | 2k               | 12k                 |

On overflow, the bridge escalates the tier (low -> mid -> high) per locked decision L4. Fails only if `high` cannot fit. **No surprise bills.**

### 4.10 3-Tier Multi-Agent Hierarchy

CleoOS enforces a strict orchestrator/lead/worker hierarchy:

| Role | Can talk to HITL? | Can call tools? | Can dispatch? |
|------|-------------------|-----------------|---------------|
| **orchestrator** | YES (the only target) | Read-only + dispatch | Leads |
| **lead** | NO | Read-only + dispatch | Own-group workers |
| **worker** | NO | Full (per permissions) | Peer queries only |

The HITL only talks to the orchestrator. Leads never execute code. Workers never dispatch other agents. Enforcement is layered:

1. **Compile time** -- cant-core lint rules TEAM-001/002/003 (581 tests)
2. **Spawn time** -- bridge refuses cross-tier spawns in the compose step
3. **Runtime** -- Pi `tool_call` hook blocks disallowed tools per role

### 4.11 Model Router

The model router (`crates/cant-router/`) selects the appropriate model for each agent spawn based on task complexity:

- **Classifier**: weighted linear scoring across 5 features (token count, syntactic complexity, reasoning depth, domain specificity, files touched). 36 tests across 6 source files.
- **Router**: rules engine with fallback chain. Enforces per-call cost caps and latency budgets.
- **Pipeline logger**: writes routing observations to `brain.db:routing_observations` for future reranker training.
- **Tiers**: `low` (Haiku-class), `mid` (Sonnet-class), `high` (Opus-class). Defined in `kind: model-routing` CANT source.

No Python. No training data required. v1 is heuristic with weighted features. v2 reranker is deferred until v1 data proves it insufficient (need 1000+ real classifications first, per L8).

### 4.12 Mental Models

Per-agent persistent mental models provide compounding intelligence across sessions:

- **Storage**: `brain.db:agents/<agent-name>/model`, scoped per `projectHash` (dynamic per project, per L5)
- **Update**: async-write via BRAIN observation queue. On agent exit, the bridge harvests session JSONL and extracts `{patterns_used, decisions_made, files_touched, outcome}`.
- **Load**: validate-on-load. Every spawn prefixes the injected mental model with a validation prompt: "Re-evaluate each claim against current code state. Mental models are dynamic per project; assume drift."
- **Decay**: entries older than `decay_after` are rejected. Weekly consolidation via `cleo brain maintenance`.
- **Reinforcement triggers**: `task_completed`, `bug_fixed` events append observations.

Mental models are never trusted blindly. Every load is a re-validation. This is the compounding loop kept honest.

### 4.13 Chat Room

The chat room (`packages/cleo-os/extensions/cleo-chatroom.ts`, 371 LOC) is the coordination primitive for multi-agent teams:

- Renders inter-agent traffic as a TUI panel via `pi.ui.setWidget()`
- Uses Pi's existing `spawnSubagent` -- no new transport, no broker
- Tails child session JSONL files for real-time message rendering
- Registers 4 tools: `send_to_lead`, `broadcast_to_team`, `report_to_orchestrator`, `query_peer`

See Section 4.2 for the full tool contract.

### 4.14 The 6 Pillars

Every CleoOS subsystem maps to one of 6 architectural pillars:

| Pillar | CleoOS Subsystem | Section |
|--------|------------------|---------|
| **Team Leads and Workers** | `kind: team` CANT + bridge enforcement + Lead tool blocking | 4.10 |
| **Expertise and Specialization** | JIT agent composer with `context_sources:` | 4.9 |
| **Agent Memory** | Per-agent BRAIN namespace + reinforcement loop + validate-on-load | 4.12 |
| **Domain Ownership** | `permissions.files: write[glob]` + runtime hook block | 4.9, 4.10 |
| **Chat Room Interface** | `cleo-chatroom.ts` Pi extension | 4.13 |
| **Configuration-Driven Harness** | The CANT source tree + model-routing CANT | Layer 2 |

Every pillar is built on primitives that ship today: PiHarness, BRAIN, CANT parser, cant-router. None require fictional engines, brokers, or new transports.

## 5. Architecture Layers

CleoOS is organized in four layers, each with a clear responsibility. See the diagram in Section 2.

| Layer | Responsibility | Key Artifacts |
|-------|---------------|---------------|
| **1. Natural-Language Surface** | Preserved injection chain: `AGENTS.md` -> `CLEO-INJECTION.md` -> `memory-bridge.md` | Generated by `cant render`, loaded by Pi |
| **2. Typed CANT SSoT** | 14 DocumentKind variants, 12 protocol files, lifecycle definitions, agent/team/tool declarations | Validated by cant-core (581 tests, 42 lint rules) |
| **3. Runtime Bridge** | Single Pi extension: compile, inject, enforce, route, compose, coordinate | `cleo-cant-bridge.ts` + `cleo-chatroom.ts` |
| **4. `cleoos` Launcher** | ~86 LOC wrapper importing `AgentSession` from Pi | `packages/cleo-os/src/cli.ts` |

Each layer depends only on the layers below it. The kernel never reaches up into CleoOS execution layers. This boundary is enforced by package dependency rules.

## 7. Design Principles

CleoOS inherits and extends the principles from `@cleocode/core`:

1. **Kernel stability** -- The kernel ships independently. CleoOS execution layers never destabilize core business logic.

2. **No new domains** -- The 10 canonical domains are fixed. CleoOS adds execution behavior on top of existing domains, never beside them.

3. **Provider neutrality** -- Every CleoOS capability works with any provider that implements the adapter contract. No provider lock-in at any layer.

4. **Local-first** -- All state lives in SQLite databases within the project's `.cleo/` directory. No cloud dependencies. Full portability.

5. **Governed execution** -- All autonomous work is bound by CANT constraints and lifecycle gates. The system does not act without governance.

6. **Memory as infrastructure** -- BRAIN is not a feature. It is the foundation that makes sustained, multi-session, multi-agent development possible.

7. **Provenance by default** -- Every action, decision, and artifact carries attribution linking it to a task, agent, session, and lifecycle stage.

8. **Progressive disclosure** -- Agents manage their own token budget. The system provides minimal responses by default and full detail on request.

## 8. What CleoOS Is Not

- CleoOS is **not a cloud platform**. It runs locally, on the developer's machine, in their terminal.
- CleoOS is **not a replacement for CLEO**. It is the full realization of what CLEO's four systems become when assembled into a complete environment.
- CleoOS is **not an AI model**. It coordinates AI agents but does not provide intelligence itself. The intelligence comes from the models; CleoOS provides memory, governance, and continuity.
- CleoOS is **not a new protocol**. It uses LAFS for response shape, CANT for control flow, and Pi's `spawnSubagent` for agent delegation. It does not invent new communication standards.
- CleoOS is **not a multi-tenant SaaS**. It is built for one developer and their agents, with NEXUS providing optional cross-project coordination.

## 9. The Operating Metaphor

If `@cleocode/core` is the kernel, then:

- **BRAIN** is the filesystem -- persistent, structured, queryable memory
- **LOOM** is the process scheduler -- work progresses through governed stages
- **LAFS** is the system call interface -- structured contracts between user space and kernel
- **NEXUS** is the network stack -- connecting isolated systems into a coherent whole
- **The JIT composer** is the init process -- the first motion that composes and starts agents
- **The chat room** is IPC -- how processes (agents) communicate
- **The Pi TUI** is the terminal -- where the operator observes and interacts
- **CANT** is the shell language -- typed instructions that the bridge interprets and executes

CleoOS is the operating system that makes all of these work together as a single, coherent development environment.
