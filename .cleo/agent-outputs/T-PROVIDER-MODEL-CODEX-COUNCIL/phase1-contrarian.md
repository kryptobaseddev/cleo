### Advisor: Contrarian

**Frame:** Assume the plan is wrong. What fails first? What's been overlooked? Why is this a worse idea than it looks?

**Evidence anchored:**
- `crates/cant-router/src/router.rs:L14-L98` — the Rust router currently returns raw model strings and downgrade re-enters the same tier route path, so a config-only change preserves the runtime shape that already cannot prove a model is invokable.
- `packages/cant/src/composer.ts:L225-L364` — the TypeScript composer mirrors the string-only tier matrix and emits `model` plus `fallbackModels` after escalation, so spawn composition has no provider/capability validation point.
- `packages/contracts/src/operations/llm.ts:L12-L59` + `packages/core/src/llm/registry.ts:L29-L170` — the actual LLM invocation layer has typed transports, base URL overrides, fallback configs, and backend factories, but the transport enum is only `anthropic | openai | gemini`.
- `packages/adapters/src/registry.ts:L49-L87` + `packages/adapters/src/index.ts:L33-L101` + `packages/caamp/providers/registry.json:L1-L523` — CLEO already has several provider registries, and they describe different things: harness adapters, bundled adapter discovery, and CAAMP tool capabilities.
- `packages/core/src/memory/llm-backend-resolver.ts:L58-L196` + `packages/core/src/metrics/model-provider-registry.ts:L1-L110` — existing dynamic lookup paths already have silent fallback behavior, short timeouts, local Ollama probing, and network-backed provider inference.
- `https://openrouter.ai/docs/guides/overview/models` + `https://vercel.com/docs/ai-gateway/models-and-providers` + `https://docs.ollama.com/api/tags` — external model catalogs change at runtime and expose different metadata surfaces, so freshness itself becomes an operational dependency.

**Findings (failure modes, from my frame only):**
1. **Stringly routing split-brain** — triggers when `.cleo/config.json` names an OpenRouter, Ollama, Vercel, or OpenAI-compatible model that is valid in a catalog but not representable by CLEO's current invocation transport set. Fails by producing a valid CANT route that reaches a spawn or LLM backend as plain `model`/`fallbackModels` strings with no matching transport, credentials, base URL, or backend factory. Detected by late `Missing API key`, `Unknown provider`, or provider SDK errors; silently when a suffix or fallback string resolves to the wrong backend.
2. **Provider namespace collision** — triggers when the shared catalog reuses existing "provider" registries without separating LLM transports from agent harness providers. Fails by accepting a provider because CAAMP or adapters say it exists, then discovering at spawn time that the provider means Claude Code, Cursor, Kimi, OpenCode, an SDK bridge, or a model marketplace rather than an invokable LLM endpoint for the selected tier. Detected by unavailable spawn support, mismatched capability flags, or fallback chains that contain providers which cannot execute the requested workload.
3. **Freshness becomes a hidden outage source** — triggers when routing depends on live or recently cached external catalogs and one source changes pricing, availability, model aliases, context windows, or times out during refresh. Fails by preserving a stale tier policy that keeps selecting an unavailable or newly expensive model, or by degrading into incomplete provider inference while the router still appears configured. Detected by repeated fallback/downgrade loops, unexpected spend or latency, and catalog refresh logs; silently when a cached model remains syntactically valid but operationally wrong.

**Verdict from this lens:** A required `RoutingConfig` by itself fails first because it moves hardcoded strings from source files into config without creating a runtime contract between tier policy, provider identity, catalog metadata, and invocation capability. A shared catalog also fails if it collapses CLEO's existing provider meanings into one namespace. The fatal risk is not hardcoding alone; it is accepting route decisions that are not provably executable by the selected transport at the moment of use.

**Single sharpest point:** RoutingConfig-only fails when a dynamically configured provider/model is catalog-valid but not invocation-valid, because CANT will emit plain model strings that cannot prove transport, credentials, base URL, fallback capability, or current availability before runtime.
