### 1. Optimizing TASKS: Context & State Management

Probabilistic agents fail when their context window gets polluted. The TASKS system shouldn't just track work; it must act as a strict **Context Firewall**.

* **Context-Window Scoping by Tier:** * **Subtask (Worker):** The agent only gets access to the specific file(s) it is editing, the immediate test file, and the atomic specification. No architecture documents allowed.
* **Task (Phase Lead):** The agent gets module-level interfaces and integration tests to aggregate worker PRs.
* **Epic (Orchestrator):** The agent sees system diagrams, ADRs, and end-to-end user journeys.
* *Optimization:* By dynamically filtering the context injected into the prompt based on the Task Tier, you drastically reduce hallucinations.


* **Cryptographic Traceability (The "+C" Expansion):** Every line of code generated should be tagged not just with the agent/human who wrote it, but with a hash of the prompt and the specific **S**pecification version that drove it. If a bug is found later, you can trace it back to a hallucination, a bad prompt, or a flawed spec.
* **State Rollback Protocol:** If a subtask fails its testing gates too many times, the system must deterministically roll back the `tasks.db` state, revert the Git branch, and flag a human. Agents should not be allowed to "flail" endlessly.

### The Cleo Testing Hierarchy (The 6-Gate System)

To support the `RCASD-IVTR+C` pipeline, Cleo expands the canonical 4 layers into 6 strict, automated gates.

#### Gate 0: Static & Security Validation (The Fast-Fail Layer)

*Before any code is executed or compiled, the harness must validate the AST (Abstract Syntax Tree) and syntax.*

* **What it is:** Linting, type-checking, cyclomatic complexity limits, and Static Application Security Testing (SAST).
* **The Agentic Purpose:** Agents often introduce unused variables, hallucinate library imports, or write overly complex functions. Gate 0 rejects these in milliseconds without wasting compute on test execution.

#### Gate 1: Unit & **Mutation** Testing (The Truth Serum)

*Tests atomic functions and classes in isolation.*

* **What it is:** Traditional Unit Testing, but heavily augmented with **Mutation Testing** (e.g., Stryker, PIT).
* **The Agentic Purpose:** *This is the most critical addition for AI.* LLMs are notorious for writing "tautological tests"—tests that just mock everything and assert `true == true`, giving a false 100% coverage. Mutation testing intentionally breaks the agent's code. If the agent's unit tests still pass, the harness flags the tests as hallucinations, rejects the commit, and deducts from the agent's reliability score.

#### Gate 2: Integration Testing

*Tests the boundaries and interfaces between modules, databases, and APIs.*

* **The Agentic Purpose:** In your "Swarming Execution Pattern" (multiple agents working in parallel waves), Integration testing is the bottleneck gate. If Agent A writes the Database schema and Agent B writes the API, the Integration tests verify the contract between them. It prevents an agent from silently changing an agreed-upon interface.

#### Gate 3: Regression Envelope (Continuous)

*Not a distinct layer, but a continuous wrapper.*

* **What it is:** Running the entire historical suite of Unit and Integration tests on the current branch.
* **The Agentic Purpose:** Agents have limited context windows. If an agent is tasked with adding a new feature to `auth.js`, it might accidentally delete the password-reset logic because it wasn't in its immediate prompt. Regression tests catch "context-window amnesia."

#### Gate 4: System & Non-Functional Testing

*Tests the holistic system, including load, stress, and security penetration.*

* **The Agentic Purpose:** Agents do not intuitively understand resource constraints. They might write an $O(N^3)$ algorithm that passes Unit tests but crashes the server under load. System testing forces the agent's code to survive under real-world pressure.

#### Gate 5: Acceptance & Invariant Testing (The Spec Check)

*Validates the implementation strictly against the 'S' (Specification) from the Planning Phase.*

* **What it is:** Behavior-Driven Development (BDD) tests (e.g., Cucumber/Gherkin) and Architectural Invariant checks.
* **The Agentic Purpose:** This ensures the agent didn't "go rogue" and invent features the user didn't ask for. If the spec says "Users can upload PDFs," and the agent also added image support because it predicted it would be helpful, the Acceptance Gate flags it as a scope violation.

### Universal Naming Conventions

To maintain strict programmatic control across any language (Python, TypeScript, Go, Rust), Cleo must enforce a rigid file, suite, and case naming taxonomy. This allows the harness to use Regex to automatically route files to the correct testing sandboxes.

#### 1. File Naming Convention

Files must explicitly declare their testing layer so the harness knows which sandbox to spin up.
**Format:** `<target_name>.test.<layer>.<extension>`

* `auth_service.test.unit.ts`
* `database_driver.test.integration.go`
* `payment_flow.test.system.py`
* `user_registration.test.acceptance.feature`

#### 2. Suite Naming Convention (The Block Level)

Test suites must map directly back to the Cleo TASKS database (`Saga -> Epic -> Task -> Subtask`) so failures can be automatically routed to the correct agent.
**Format:** `[TaskID] - [Component/Feature Name]`

* *Example:* `[T-492] - AuthenticationService`
* *Example:* `[E-12] - User Checkout Flow`

#### 3. Case Naming Convention (The 'Should' Standard)

Agent-generated test cases must follow a strict behavioral naming pattern. This prevents vague tests like `test_auth_1()`. It forces the agent to define the *State*, the *Action*, and the *Expected Outcome*.
**Format:** `should <expected outcome> when <state/action occurs>`

* *Example (Unit):* `should throw InvalidTokenError when expired JWT is provided`
* *Example (Integration):* `should rollback database transaction when payment API returns 500`
* *Example (Acceptance):* `should grant dashboard access when user completes 2FA challenge`

### The Agentic TDD Rule (Strict Enforcement)

To make Cleo truly immutable, you must enforce **Agentic Shift-Left TDD**.

1. During the **Specification** phase, an Orchestrator Agent writes the *Acceptance and Integration* tests based on the spec.
2. These tests are committed to the branch and will currently **FAIL**.
3. The Subtask (Implementation) Agent is then spawned. Its only objective is to write code until those pre-existing tests pass.
4. The Implementation agent is *never* allowed to modify the Orchestrator's tests.

By separating the "Test Writer" agent from the "Implementation" agent, you create an adversarial dynamic that mathematically guarantees the specification is met without hallucinations.

### The 4-Step Agentic TDD Protocol

To make the Cleo SDLC immutable, this process must be strictly enforced:

1. **The Specification Phase (Test Generation):** An Orchestrator Agent is tasked with writing the Acceptance and Integration tests based directly on the project's specification.
2. **The "Red" Phase (Failing State):** These generated tests are committed to the branch. Because the actual feature hasn't been written yet, these tests will currently FAIL.
3. **The Implementation Phase (Subtask Execution):** A separate Subtask (Implementation) Agent is then spawned. Its sole objective and stopping condition is to write code until those pre-existing, failing tests turn green.
4. **The Immutable Rule:** The Implementation agent is **never** allowed to modify or delete the Orchestrator's tests.

### Test Formatting and Configuration

For LLMs to excel in this adversarial setup, standard unit test names like `test_auth_1()` are too vague and allow the Implementation Agent too much creative liberty.

Instead, the Orchestrator Agent must configure tests using a strict **State, Action, Expected Outcome** format (similar to Gherkin's Given/When/Then), strictly adhering to the `should <outcome> when <state/action occurs>` naming convention.

**Examples of LLM-Favorable Test Stubs:**

* **Unit Test:** `should throw InvalidTokenError when expired JWT is provided`
* **Integration Test:** `should rollback database transaction when payment API returns 500`
* **Acceptance Test:** `should grant dashboard access when user completes 2FA challenge`

By establishing these executable test stubs *first*, the Orchestrator sets up rigid gates. The Implementation agent is then forced into a deterministic recursive loop: write code, run tests, fail, fix code, run tests, pass.
