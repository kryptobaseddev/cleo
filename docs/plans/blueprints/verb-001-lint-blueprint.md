# VERB-001 Lint Rule Blueprint (deferred from Wave 0)

**Status**: READY FOR PICKUP
**Source**: feature-dev:code-architect agent (session 2026-04-08, follow-up for VERB-STANDARDS clarification)
**Target wave**: Wave 5 or Wave 6 (paired with the model router or the JIT composer when the bridge has concrete need for verb validation)
**Canonical plan**: `docs/plans/CLEO-ULTRAPLAN.md` (VERB-001 explicitly deferred per architect Option C)
**Dependencies**: Wave 0.5 grammar additions should ship first; VERB-001 walks AST nodes that include the new `team`/`tool` blocks and agent `context_sources`/`mental_model` sub-blocks

This blueprint captures the VERB-001 lint rule design for validating that CANT statement bodies reference only canonical CLEO verbs from the CLI dispatch registry. Implementation is **deferred from Wave 0** — the minimum grammar additions shipped without it to keep Wave 0 surgical and because the TS-registry extraction adds meaningful complexity.

---

## 1. Context: The Namespace Split

**VERB-STANDARDS.md** governs runtime operation verb names exposed via CLI dispatch (e.g., `tasks.add`, `memory.find`, `orchestrate.spawn`). It is stale in places because it still references MCP operations — MCP was removed 2026-04-04 per `memory/mcp-removal-complete.md`. The **executable source of truth** is `packages/cleo/src/dispatch/registry.ts` (CLI-only, authoritative). VERB-001 reads from the registry, NOT from the markdown.

**CANT grammar** governs DSL section keywords in `.cant` source files. These are syntactically and semantically orthogonal to runtime verbs. The one intersection: operation-style tokens (`<domain>.<action>`) that appear INSIDE CANT statement bodies (hooks, workflows, tool implementation properties, constraint bodies) should reference canonical verbs.

**Zero namespace collision** between the Wave 0 grammar keywords (`team`, `tool`, `role`, `tier`, `context_sources`, `mental_model`, plus all JIT sub-keys) and the VERB-STANDARDS enforced matrix — verified during architect dispatch.

## 2. Architecture Decision

Build-time extraction from `packages/cleo/src/dispatch/registry.ts` (Option A from the architect dispatch). Rationale:
- `cant-core` is a pure Rust crate with no TS runtime dependency
- `build.rs` already reads files across the monorepo (existing `hook-mappings.json` pattern at `generated/events.rs`)
- Parsing `operation: 'foo'` string literals from `registry.ts` with regex is lower coupling than parsing the full TS AST
- Output: a sorted `&[(&str, &str)]` embedded in `src/generated/verbs.rs` — O(log n) binary-search lookup at validation time
- No new Cargo dependencies (binary search in `std`)

**Rejected options**:
- **Option B** (hardcode + CI diff against registry.ts): still requires manual updates, opens drift risk
- **Option C** (defer indefinitely): chosen for Wave 0 but not as the permanent answer

**VERB-001 severity**: `Warning` not `Error`. Rationale: operation-style tokens might be CANT variable names (edge case). Warn, don't break. A future `VERB-002` at Error severity can be introduced once the corpus is established and patterns are stable.

## 5. Generated Code Shape (`src/generated/verbs.rs`)

```rust
// @generated — DO NOT EDIT.
// Source: packages/cleo/src/dispatch/registry.ts
// Regenerated by build.rs on every build.

/// Sorted array of canonical CLEO verb operations.
/// Each entry is `(domain, operation)` where operation may contain dots
/// for namespaced actions (e.g. `("pipeline", "manifest.append")`).
pub const CANONICAL_VERB_OPS: &[(&str, &str)] = &[
    ("admin", "dash"),
    ("admin", "doctor"),
    // ... ~200 entries extracted from registry.ts ...
    ("tasks", "add"),
    ("tasks", "complete"),
    ("tasks", "find"),
    ("tasks", "show"),
    ("tasks", "update"),
    // ...
];

/// The 10 canonical CLEO domains. Used by the walker to decide which
/// tokens are candidates for VERB-001 enforcement.
pub const CANONICAL_DOMAINS: &[&str] = &[
    "tasks", "memory", "session", "pipeline", "orchestrate",
    "check", "admin", "nexus", "tools", "sticky",
];

/// Returns `true` iff `(domain, op)` is a canonical CLEO verb operation.
/// Uses binary search — O(log n).
pub fn is_canonical_verb_op(domain: &str, op: &str) -> bool {
    CANONICAL_VERB_OPS
        .binary_search(&(domain, op))
        .is_ok()
}
```

Add a second generation phase alongside the existing `events.rs` generation:

```rust
fn generate_verbs() -> Result<(), Box<dyn std::error::Error>> {
    let registry_path = Path::new("../../packages/cleo/src/dispatch/registry.ts");
    println!("cargo:rerun-if-changed={}", registry_path.display());

    let source = fs::read_to_string(registry_path)?;

    // Two-pass regex extraction:
    // 1. Find every object literal of shape `{ ... domain: 'X', ... operation: 'Y' ... }`
    // 2. For each object, extract (domain, operation) as a tuple
    //
    // Pattern: look for `domain: '(\w+)'` and `operation: '([\w.]+)'` within
    // each `OperationDef` object boundary. Registry entries are whitespace-
    // delimited objects in a const `OPERATIONS: OperationDef[] = [...]`.

    let domain_re = regex::Regex::new(r"domain:\s*['"](\w+)['"]")?;
    let op_re = regex::Regex::new(r"operation:\s*['"]([\w.]+)['"]")?;

    // Scan object boundaries and pair (domain, operation) within each
    let mut pairs: Vec<(String, String)> = Vec::new();
    // ... extraction logic ...

    pairs.sort();
    pairs.dedup();

    // Write generated file
    let mut out = String::new();
    out.push_str("// @generated — DO NOT EDIT.\n");
    out.push_str("// Source: packages/cleo/src/dispatch/registry.ts\n\n");
    out.push_str("pub const CANONICAL_VERB_OPS: &[(&str, &str)] = &[\n");
    for (d, o) in &pairs {
        out.push_str(&format!("    ({:?}, {:?}),\n", d, o));
    }
    out.push_str("];\n\n");
    out.push_str("pub const CANONICAL_DOMAINS: &[&str] = &[\n");
    out.push_str("    \"tasks\", \"memory\", \"session\", \"pipeline\", \"orchestrate\",\n");
    out.push_str("    \"check\", \"admin\", \"nexus\", \"tools\", \"sticky\",\n");
    out.push_str("];\n\n");
    out.push_str("pub fn is_canonical_verb_op(domain: &str, op: &str) -> bool {\n");
    out.push_str("    CANONICAL_VERB_OPS.binary_search(&(domain, op)).is_ok()\n");
    out.push_str("}\n");

    write_if_changed(Path::new("src/generated/verbs.rs"), &out)?;
    Ok(())
}
```

**Note**: the existing build.rs may not yet use `regex`. If it does not, either add `regex = "1"` to `[build-dependencies]` in `Cargo.toml`, or implement the extraction with manual string scanning (slower to write but zero dependency cost).

**Important about exceptions**: VERB-STANDARDS §5 documents live-but-legacy operations (`nexus.query`, `admin.fix`, `orchestrate.bootstrap`). These appear naturally in `registry.ts` because they're live operations — they will be in `CANONICAL_VERB_OPS` and pass VERB-001 automatically. No special exception handling needed in the extractor.

## 7. `validate/verbs.rs` Walk Algorithm

```rust
//! VERB-001 lint rule: canonical verb compliance.
//!
//! Validates that operation-style tokens in CANT statement bodies
//! reference canonical CLEO verbs from the CLI dispatch registry.
//! The canonical verb list is extracted from
//! `packages/cleo/src/dispatch/registry.ts` at build time.

use crate::dsl::ast::*;
use crate::generated::verbs::{is_canonical_verb_op, CANONICAL_DOMAINS};
use super::context::ValidationContext;
use super::diagnostic::Diagnostic;

pub fn check_all(doc: &CantDocument, _ctx: &ValidationContext) -> Vec<Diagnostic> {
    let mut diags = Vec::new();
    for section in &doc.sections {
        match section {
            Section::Agent(agent) => walk_agent(agent, &mut diags),
            Section::Hook(hook) => walk_hook(hook, &mut diags),
            Section::Workflow(w) => walk_workflow(w, &mut diags),
            Section::Pipeline(p) => walk_pipeline(p, &mut diags),
            Section::Tool(t) => walk_tool(t, &mut diags),
            _ => {}
        }
    }
    diags
}

fn walk_agent(agent: &AgentDef, diags: &mut Vec<Diagnostic>) {
    // Walk hook bodies, constraint blocks, and context_sources properties
    for hook in &agent.hooks {
        scan_statements(&hook.body, diags);
    }
    // context_sources and mental_model are Vec<Property> — scan values
    scan_properties(&agent.context_sources, diags);
    scan_properties(&agent.mental_model, diags);
}

fn walk_tool(tool: &ToolDef, diags: &mut Vec<Diagnostic>) {
    // Look for `implementation:` property and scan its value for verb tokens
    for prop in &tool.properties {
        if prop.key.value == "implementation" {
            scan_property_value(&prop.value, prop.value.span(), diags);
        }
    }
}

fn scan_properties(props: &[Property], diags: &mut Vec<Diagnostic>) {
    for prop in props {
        scan_property_value(&prop.value, prop.value.span(), diags);
    }
}

fn scan_property_value(value: &Value, span: Span, diags: &mut Vec<Diagnostic>) {
    // Extract string representation and scan for domain.action patterns
    let text = match value {
        Value::String(sv) => sv.raw.clone(),
        Value::Identifier(id) => id.clone(),
        _ => return,
    };

    // Look for <identifier>.<identifier> or <identifier>.<namespace>.<action>
    // where the root segment is a canonical CLEO domain
    for token in scan_tokens(&text) {
        if let Some((domain, op)) = parse_domain_op(&token) {
            if CANONICAL_DOMAINS.contains(&domain) && !is_canonical_verb_op(domain, op) {
                diags.push(Diagnostic::warning(
                    "VERB-001",
                    format!(
                        "Non-canonical verb '{}.{}'. \
                         Use the canonical CLI dispatch verb from \
                         packages/cleo/src/dispatch/registry.ts \
                         (see docs/specs/VERB-STANDARDS.md §2 for the enforced matrix).",
                        domain, op
                    ),
                    span,
                ));
            }
        }
    }
}

fn scan_tokens(text: &str) -> Vec<String> {
    // Simple scanner: split on whitespace and punctuation (except `.` and `_`),
    // return any token containing at least one `.` and matching
    // [a-z_][a-z0-9_]*(\.[a-z_][a-z0-9_]*)+
    // Use a regex or manual scan — implementation detail
    todo!()
}

fn parse_domain_op(token: &str) -> Option<(&str, &str)> {
    // Split on first `.` — everything before is the domain, everything after is the op
    let dot_idx = token.find('.')?;
    let domain = &token[..dot_idx];
    let op = &token[dot_idx + 1..];
    Some((domain, op))
}

fn walk_hook(hook: &HookDef, diags: &mut Vec<Diagnostic>) { scan_statements(&hook.body, diags); }
fn walk_workflow(w: &WorkflowDef, diags: &mut Vec<Diagnostic>) { /* ... */ }
fn walk_pipeline(p: &PipelineDef, diags: &mut Vec<Diagnostic>) { /* ... */ }

fn scan_statements(statements: &[Statement], diags: &mut Vec<Diagnostic>) {
    for stmt in statements {
        match stmt {
            Statement::Property(p) => scan_property_value(&p.value, p.span, diags),
            Statement::Expression(e) => { /* scan expression text */ }
            _ => {}
        }
    }
}
```

## 10. Critical Details

### `nexus.query` and other live exceptions

VERB-STANDARDS §5 lists legacy operations that use non-canonical verbs but are live in the registry: `nexus.query`, `admin.fix`, `orchestrate.bootstrap`. These WILL appear in `CANONICAL_VERB_OPS` because they're in `registry.ts`. VERB-001 will pass them automatically. No special-case code needed.

### Operation names with dots (namespaced actions)

Many operations have namespaced action parts: `pipeline.manifest.append`, `pipeline.stage.validate`, `check.schema.validate`. The tuple stored is `(domain, full_operation_string)`, e.g., `("pipeline", "manifest.append")`. The walker detects these as three-segment tokens and tests whether `(segments[0], segments[1..].join("."))` matches.

### Pi tool names vs. CANT tool block names

`tool` blocks in CANT declare Pi tool bindings. The tool NAME (e.g., `dispatch_worker`, `send_to_lead`) follows Pi's snake_case convention and is NOT a VERB-STANDARDS verb. VERB-001 does NOT flag the tool name itself — it only walks property bodies inside the tool block, specifically the `implementation:` property value.

### Severity is Warning, not Error

Rationale: CANT variable names or string literals might legitimately contain `domain.action` patterns (e.g., documentation strings). Until the corpus is established, VERB-001 warns without blocking. A future `VERB-002` at Error severity can be introduced later when confidence is high.

### Regex vs manual scanning

If `build.rs` uses `regex`, add `regex = "1"` to `[build-dependencies]` in `crates/cant-core/Cargo.toml`. If the team prefers zero build dependencies, a simple manual scanner for `operation: 'foo'` substrings works — less elegant but dependency-free. Look at how `build.rs` currently parses `hook-mappings.json` for prior art.
