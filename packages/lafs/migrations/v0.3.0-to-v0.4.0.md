# Migration Guide: v0.3.0 to v0.4.0 (Envelope Rationalization)

## Summary

LAFS v0.4.0 rationalizes the canonical envelope by making null-valued optional fields truly optional (omittable), enforcing strict mode semantics, and introducing extension points for vendor metadata and advisory warnings.

**Breaking changes:** 6
**New features:** 2 (`_extensions`, `_meta.warnings`)

---

## Breaking Changes

### 1. `page` field is now optional

**What changed:** In v0.3.0, the `page` field was required on every envelope, even when the operation did not involve pagination. In v0.4.0, `page` MAY be omitted entirely when not applicable.

**Before (v0.3.0):** `page` required, set to `null` for non-paginated responses.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.3.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "user.create",
    "requestId": "req_101",
    "transport": "http",
    "strict": true,
    "mvi": true,
    "contextVersion": 0
  },
  "success": true,
  "result": { "id": "usr_42", "name": "Ada Lovelace" },
  "error": null,
  "page": null
}
```

**After (v0.4.0):** `page` omitted for non-paginated responses.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.4.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "user.create",
    "requestId": "req_101",
    "transport": "http",
    "strict": true,
    "mvi": true,
    "contextVersion": 0
  },
  "success": true,
  "result": { "id": "usr_42", "name": "Ada Lovelace" }
}
```

**Action required:**
- Remove `"page": null` from all non-paginated response envelopes.
- Update any code that unconditionally accesses `.page` — check for the field's presence before reading it.
- Paginated responses SHOULD still include the full `page` object.

### 2. `error` field is now optional when `success=true`

**What changed:** In v0.3.0, the `error` field was required on every envelope, set to `null` on success. In v0.4.0, `error` MAY be omitted when `success=true`. It is still **required** when `success=false`.

**Before (v0.3.0):** `error` required, explicitly set to `null` on success.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.3.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "item.get",
    "requestId": "req_200",
    "transport": "cli",
    "strict": true,
    "mvi": true,
    "contextVersion": 0
  },
  "success": true,
  "result": { "id": "item_7", "title": "Widget" },
  "error": null,
  "page": null
}
```

**After (v0.4.0):** `error` omitted on success.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.4.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "item.get",
    "requestId": "req_200",
    "transport": "cli",
    "strict": true,
    "mvi": true,
    "contextVersion": 0
  },
  "success": true,
  "result": { "id": "item_7", "title": "Widget" }
}
```

**Action required:**
- Remove `"error": null` from success envelopes.
- Update consumers to check `success === false` before accessing `.error`, rather than assuming the field exists.
- Error envelopes (`success=false`) MUST still include the full `error` object.

### 3. `_extensions` field added

**What changed:** A new optional `_extensions` object is now a declared property in the envelope schema. It allows vendor-specific metadata without violating strict mode (since it is a declared property, not an additional property).

**Before (v0.3.0):** No standard way to attach vendor metadata. Any extra top-level fields would violate strict mode.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.3.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "order.submit",
    "requestId": "req_301",
    "transport": "http",
    "strict": true,
    "mvi": true,
    "contextVersion": 1
  },
  "success": true,
  "result": { "orderId": "ord_99" },
  "error": null,
  "page": null,
  "x-trace-id": "abc-123"
}
```

The above envelope would **fail validation** in v0.3.0 strict mode because `x-trace-id` is an undeclared top-level property.

**After (v0.4.0):** Use `_extensions` for vendor metadata.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.4.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "order.submit",
    "requestId": "req_301",
    "transport": "http",
    "strict": true,
    "mvi": true,
    "contextVersion": 1
  },
  "success": true,
  "result": { "orderId": "ord_99" },
  "_extensions": {
    "x-trace-id": "abc-123",
    "x-region": "us-east-1"
  }
}
```

**Action required:**
- Move any vendor-specific top-level fields into the `_extensions` object.
- Use the `x-` prefix convention for extension keys.
- Consumers MUST NOT rely on extension fields for protocol-required behavior.
- No action needed if you were not using custom top-level fields.

### 4. `_meta.warnings` array added

**What changed:** A new optional `warnings` array is available inside `_meta` for deprecation notices, policy advisories, and migration hints. Each warning object requires `code` and `message`; optional fields include `deprecated`, `replacement`, and `removeBy`.

**Before (v0.3.0):** No mechanism for non-fatal warnings. Deprecation notices had to be communicated out-of-band (documentation, logs, etc.).

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.3.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "report.generate",
    "requestId": "req_400",
    "transport": "sdk",
    "strict": true,
    "mvi": true,
    "contextVersion": 0
  },
  "success": true,
  "result": { "reportUrl": "https://example.com/reports/42" },
  "error": null,
  "page": null
}
```

**After (v0.4.0):** Inline warnings attached to the response.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.4.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "report.generate",
    "requestId": "req_400",
    "transport": "sdk",
    "strict": true,
    "mvi": true,
    "contextVersion": 0,
    "warnings": [
      {
        "code": "DEPRECATED_FIELD",
        "message": "The 'reportUrl' field is deprecated. Use 'result.download.href' instead.",
        "deprecated": "result.reportUrl",
        "replacement": "result.download.href",
        "removeBy": "1.0.0"
      }
    ]
  },
  "success": true,
  "result": { "reportUrl": "https://example.com/reports/42" }
}
```

**Action required:**
- Producers: Start emitting `_meta.warnings` for any deprecated fields or behaviors.
- Consumers: Check for `_meta.warnings` and surface them to users or logs. Do not treat warnings as errors — they are advisory.
- No action needed if you have no deprecations to announce.

### 5. Strict mode enforced — `strict:true` rejects additional properties

**What changed:** In v0.3.0, `strict:true` was declared in `_meta` but was not enforced by the schema — additional top-level properties were silently accepted. In v0.4.0, when `_meta.strict` is `true`, the schema applies `additionalProperties: false` at the envelope level. Only the declared properties (`$schema`, `_meta`, `success`, `result`, `error`, `page`, `_extensions`) are permitted. When `strict:false`, additional properties are allowed.

**Before (v0.3.0):** Extra fields silently accepted even with `strict:true`.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.3.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "task.list",
    "requestId": "req_500",
    "transport": "cli",
    "strict": true,
    "mvi": true,
    "contextVersion": 0
  },
  "success": true,
  "result": { "tasks": [] },
  "error": null,
  "page": null,
  "debugInfo": { "queryMs": 42 },
  "customField": "allowed-silently"
}
```

This passed validation in v0.3.0 despite `strict:true`.

**After (v0.4.0):** Same envelope **fails validation** with a schema error.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.4.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "task.list",
    "requestId": "req_500",
    "transport": "cli",
    "strict": true,
    "mvi": true,
    "contextVersion": 0
  },
  "success": true,
  "result": { "tasks": [] },
  "debugInfo": { "queryMs": 42 },
  "customField": "this-will-be-rejected"
}
```

The above will produce a validation error because `debugInfo` and `customField` are not declared envelope properties.

**Corrected v0.4.0 envelope:** Move custom data into `_extensions` or `result`.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.4.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "task.list",
    "requestId": "req_500",
    "transport": "cli",
    "strict": true,
    "mvi": true,
    "contextVersion": 0
  },
  "success": true,
  "result": { "tasks": [] },
  "_extensions": {
    "x-debug-query-ms": 42
  }
}
```

**Action required:**
- Audit all envelope producers for undeclared top-level fields.
- Move vendor/debug data into `_extensions`.
- Move domain data into `result`.
- Alternatively, set `strict:false` if additional properties are intentionally needed (not recommended for production).

### 6. Strict mode behavior — omit optional null fields instead of setting them to null

**What changed:** In v0.4.0, when `strict:true`, optional fields that would be `null` (such as `error` on success, `page` on non-paginated responses) SHOULD be **omitted entirely** rather than explicitly set to `null`. This produces smaller, cleaner envelopes and aligns with the MVI principle.

**Before (v0.3.0):** Optional null fields explicitly included.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.3.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "config.update",
    "requestId": "req_600",
    "transport": "http",
    "strict": true,
    "mvi": true,
    "contextVersion": 3
  },
  "success": true,
  "result": { "setting": "theme", "value": "dark" },
  "error": null,
  "page": null
}
```

**After (v0.4.0):** Null optional fields omitted in strict mode.

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.4.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "config.update",
    "requestId": "req_600",
    "transport": "http",
    "strict": true,
    "mvi": true,
    "contextVersion": 3
  },
  "success": true,
  "result": { "setting": "theme", "value": "dark" }
}
```

**Non-strict mode (v0.4.0):** When `strict:false`, including `"error": null` and `"page": null` is still permitted (but not required).

```json
{
  "$schema": "https://lafs.dev/schemas/v1/envelope.schema.json",
  "_meta": {
    "specVersion": "0.4.0",
    "schemaVersion": "1.0.0",
    "timestamp": "2026-02-11T00:00:00Z",
    "operation": "config.update",
    "requestId": "req_601",
    "transport": "http",
    "strict": false,
    "mvi": true,
    "contextVersion": 3
  },
  "success": true,
  "result": { "setting": "theme", "value": "dark" },
  "error": null,
  "page": null
}
```

**Action required:**
- For `strict:true` producers: stop emitting `"error": null` and `"page": null` on success envelopes.
- For consumers: use defensive access patterns (e.g., `envelope.error ?? null`, optional chaining) rather than assuming these fields exist.

## Backward Compatibility Notes

| Aspect | Detail |
|---|---|
| Schema version | `schemaVersion` remains `"1.0.0"` — the envelope schema ID has not changed. |
| Spec version | Update `specVersion` from `"0.3.0"` to `"0.4.0"` in all envelopes. |
| Old envelopes with null fields | Envelopes that still include `"error": null` or `"page": null` remain **valid** under the v0.4.0 schema (the fields are optional, not forbidden). However, strict-mode producers SHOULD omit them. |
| `_extensions` | Fully optional. Existing envelopes without `_extensions` are valid. |
| `_meta.warnings` | Fully optional. Existing envelopes without `warnings` are valid. |
| `strict:false` | Envelopes with `strict:false` are unaffected by the additionalProperties enforcement. |

**Key takeaway:** The v0.4.0 changes are **relaxations** (making fields optional) and **additions** (new optional fields), except for strict mode enforcement which is a **tightening**. Most existing v0.3.0 envelopes will continue to validate unless they relied on undeclared top-level properties under `strict:true`.

## New Features

### `_extensions` — Vendor Extension Object

The `_extensions` field provides a standard location for vendor-specific metadata that is outside the LAFS protocol scope.

```json
{
  "_extensions": {
    "x-myvendor-trace-id": "trace_abc123",
    "x-myvendor-region": "eu-west-1",
    "x-billing-cost-units": 15
  }
}
```

**Rules:**
- Keys SHOULD use the `x-` prefix convention.
- `_extensions` is a declared property, so it is permitted even when `strict:true`.
- Consumers MUST NOT rely on extension fields for protocol-required behavior.
- Producers MAY omit `_extensions` entirely.

### `_meta.warnings` — Advisory Warnings Array

The `warnings` array enables producers to communicate deprecation notices, policy changes, and migration hints inline with the response.

```json
{
  "_meta": {
    "warnings": [
      {
        "code": "DEPRECATED_FIELD",
        "message": "The 'legacyId' field will be removed in v1.0.0. Use 'id' instead.",
        "deprecated": "result.legacyId",
        "replacement": "result.id",
        "removeBy": "1.0.0"
      },
      {
        "code": "RATE_LIMIT_APPROACHING",
        "message": "You have used 95% of your hourly quota."
      }
    ]
  }
}
```

**Warning object fields:**

| Field | Required | Description |
|---|---|---|
| `code` | Yes | Warning identifier (e.g., `DEPRECATED_FIELD`, `RATE_LIMIT_APPROACHING`) |
| `message` | Yes | Human-readable description |
| `deprecated` | No | The deprecated feature or field name |
| `replacement` | No | Suggested replacement |
| `removeBy` | No | Version when the deprecated feature will be removed |

## Conformance Tier Implications

| Tier | Impact |
|---|---|
| **Core** | Minimal impact. Envelopes that omit `error` and `page` on success still validate against `envelope.schema.json`. The `envelope_invariants` check is unaffected. |
| **Standard** | No new checks added. `meta_strict_present` and `meta_mvi_present` checks are unchanged. |
| **Complete** | Strict mode enforcement now has teeth — implementations that previously passed with undeclared top-level fields under `strict:true` MUST remove them or move them to `_extensions`. |

## Migration Checklist

- [ ] Update `specVersion` to `"0.4.0"` in all envelope producers
- [ ] Remove `"error": null` from success envelopes (required for strict mode, recommended otherwise)
- [ ] Remove `"page": null` from non-paginated envelopes (required for strict mode, recommended otherwise)
- [ ] Audit for undeclared top-level properties in `strict:true` envelopes — move to `_extensions` or `result`
- [ ] Update consumers to handle missing `error` and `page` fields defensively
- [ ] Adopt `_extensions` for any vendor-specific metadata (optional)
- [ ] Adopt `_meta.warnings` for deprecation notices (optional)
- [ ] Run schema validation against `schemas/v1/envelope.schema.json` to verify compliance
